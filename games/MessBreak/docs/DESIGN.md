# MessBreak 企画・設計

MessBreak 固有の企画・設計・開発記録はこの `games/MessBreak/docs/` に置く。
Statee(フレームワーク)側の `docs/adr/` には持ち込まない。
Statee の ADR に書くのは、`src/` や repo 構成に波及する決定だけ。

## コンセプト

- 2D 見下ろし・リアルタイム戦闘のダンジョンアドベンチャー
- **ツインスティック型(移動とエイムの独立)+スキル主体、射撃は繋ぎ**。
  TPS の基本である「敵の攻撃を移動・回避でかわしながら、エイムは置いて維持する」手触りを
  2D 見下ろしで再現する。移動軸(WASD)と照準軸(マウス/右スティック)を分離し、
  ドッジ中もエイムが維持されることをコアにする。
  倒す主役はクールダウン制のスキルとその組み合わせで、通常射撃は低ダメージの繋ぎ
  (リズムとエイム維持の理由づけ)
- 敵は物量(バンサバ型)ではなく、少数の敵の攻撃パターンと操作の手触りで面白さを作る。
  敵が動く(横移動・射線回避・突進)ことで偏差射撃・置きエイムに意味を持たせる
- 内部は完全 2D 座標。見た目の奥行きは Godot の Y ソートで表現する(疑似 2.5D)
- キャラクター3人を切り替えて戦う。3人とも 18 歳の美少女:
  - アタッカー
  - バッファー(ヒーラー)
  - デバッファー

  キャラ切り替え=スキルセット切り替え。切り替えを挟むスキルコンボ
  (例: デバッファーで足止め → アタッカーの大技を置きエイムで全弾当てる)が爽快感の中心

## 名前

- タイトルは **MessBreak**(表記はキャメルケースで統一)。
  「Mess(混沌・敵の群れ)を Break(打ち破る)」
- スラング上の問題・既存タイトルとの衝突がないことは確認済み(2026-07)。
  近縁名として Steam の「MESS」「Mecha BREAK」が存在する点のみ留意

## 開発方針

- 境界の掟(docs/USING.md)に従い、ルール・移動・当たり判定・敵 AI は
  `MessBreak.Logic`(純C#)に置き、xUnit + Shouldly でテストする。
  Godot 層は描画・入力・Statee 配線のみ
- 物理・衝突は Godot 物理に頼らず、Logic 層の自前判定(円・矩形)で行う
- 乱数は seed 注入、進行は tick 駆動で決定的に保つ
- ゲームバランスに関わる数値(速度・HP・攻撃力・部屋サイズ等)は
  ハードコードせず Logic 層の設定オブジェクトに集約する

## 最初の縦切り(スコープ)

「部屋1つ + 敵1種 + プレイヤーの攻撃と回避」

1. プレイヤーの移動(部屋の矩形内)
2. 敵1種(索敵 → 接近 → 予備動作 → 攻撃 → 硬直 のステートマシン)
3. プレイヤーの攻撃(近接)と回避(ドッジ)
4. 勝敗判定(敵撃破 / プレイヤー HP 0)

タイトル画面・リザルト・部屋遷移は縦切りの後。

### 縦切り1の到達点(2026-07-16)

- `BattleLogic`(tick 駆動・決定論)として実装済み。ユニットテスト 21 件緑
- Godot 層は `_PhysicsProcess`(60Hz)で 1 tick。人間はキーボード
  (矢印/WASD 移動、Z/J 攻撃、X/K ドッジ)、エージェントは freeze + `tick` コマンド
  (`--arg frames=N,input=right+attack`)でプレイする
- headless E2E で「接近 → 攻撃 → 敵の予備動作をドッジ回避 → 撃破 → Victory」を確認済み
- 描画はプレースホルダ(円と HUD 文字)。スプライト(art/)への差し替えは今後

## 縦切り2(ツインスティック化)

1. **エイムの分離+通常射撃**: 近接攻撃を廃し、AimDir(移動と独立)+弾の射撃に置き換える。
   移動・ドッジしながら撃てる/エイムを維持できることが受け入れ条件
2. スキル1個(クールダウン+範囲ダメージ)
3. キャラ切り替え(スキルセットの差し替え)

2・3 は 1 の完了後に着手。

### 縦切り2-2 の到達点(2026-07-17)

- スキル1個を実装: **E / Q キー**で範囲ダメージ(半径 SkillRadius 40・SkillDamage 3 =
  的を一撃)。クールダウン 180 tick
- **爆心はレティクル(カーソル)位置**(触っての要望で「向いている方向の固定距離」から変更)。
  射程 SkillRange(80)を超える指定はその方向の射程上限にクランプ。
  照準点を持たないデバイス(ゲームパッド)は TickInput.AimPoint が null になり、
  向いている方向の射程いっぱいへのフォールバック
- 発動はドッジ中不可・移動と両立
- 命中統計(ShotCount/HitCount)は射撃専用のままにし、スキル命中は数えない(KillCount は共通)
- SkillBurst イベント → Godot 層が爆発リング+専用効果音(audio/skill.sfx.txt)に翻訳
- エージェントは `tick` の入力トークン `skill` で発動。爆心の絶対座標は
  `skillx`/`skilly`(例: `input=skill,skillx=480,skilly=180`。省略でフォールバック)。
  headless E2E で「スプリント接近 → 置きエイム → スキル一撃 → リスポーン開始」を確認済み
- 遊びの核(キャラ切り替えを挟むスキルコンボ)の検証はこのスキルを土台に、
  次の縦切り2-3(キャラ切り替え)で行う

### 縦切り2-3 の到達点(2026-07-17): キャラ切り替え+コンボの最小検証

遊びの核「切り替えを挟むスキルコンボ」を最小構成で実装:

- キャラ2人: **アタッカー**(範囲ダメージ 3)と**デバッファー**(被ダメージ2倍デバフ・
  持続 300 tick・ダメージなし)。スキルの操作(E/Q・レティクル爆心)は共通
- 切り替えは**直接指定**: **1 キー=アタッカー、2 キー=デバッファー**(トグルではない。
  3人目の追加も 3 キーで自然に拡張できる)。切り替えクールダウン 30 tick。
  操作中キャラの再指定は不発扱い(クールダウンを消費しない)
- **スキルクールダウンはキャラごとに独立**(切り替えの存在意義)
- バランス値はキャラ単位の `CharacterConfig`(スキル CD・射程・半径・効果量)に分離。
  `BattleConfig.Attacker / .Debuffer` で個別調整でき、3人目はキャラ定義1個の追加で済む。
  デバフ倍率は付与時点のデバッファー設定値を保持する(付与後にキャラを切り替えても変わらない)
- 的の HP を 6 に変更: アタッカースキル単発(3)では倒れず、
  **コンボ(デバフ→切り替え→大技=3×2=6)でのみ一撃**。これがコンボの物差し
- デバフは足止めではなくダメージ増幅にした(的が動かない射撃場では足止めに意味がないため。
  足止め系は敵が動くようになったら検討)。リスポーンした的はデバフを引き継がない
- 見た目: デバッファー操作中はスプライトを青緑ティント(専用スプライトまでの色違い)。
  デバフ中の的は紫リング。HUD に操作キャラ名と両スキルの CD を表示
- エージェントは `tick` の入力トークン `char1` / `char2` で切り替え。
  headless E2E で「接近 → char2 → デバフ → char1 → 大技一撃(HP6→0)」を確認済み

### 縦切り2-1 の到達点(2026-07-16)

- 近接攻撃を廃し、ツインスティック射撃に置き換え済み。ユニットテスト 29 件緑
- エイム(AimDir)は移動と独立。移動・撃ちながらエイム維持、ドッジは移動方向へ
  (移動入力なしでは不発。2026-07-17 変更)、ドッジ中は射撃不可
- 弾は安定 ID 付き。クールダウン制。敵命中と部屋外で消滅
- 入力: 移動 WASD/矢印、エイム=マウスカーソル、射撃=左クリック/Z/J、
  ドッジ=Space、スプリント=左Shift。右クリックは将来のスキル発動用に空けてある
  (マウスエイムでは「照準ボタン」は仕事がなく、ADS やロックオンを入れない限り不要と判断)。
  エージェントは `tick` コマンドに `aimx`/`aimy` 引数を追加
  (例: `--arg frames=60,input=fire+sprint,aimx=160,aimy=-45`)
- 向きの可視化: 自機・敵とも円+ノーズ(短い銃身)。自機はさらに細い照準線。
  敵のノーズはプレイヤーへの視線(Idle 中は非表示)
- headless E2E で「下移動しながら右エイム(独立確認)→斜めの置きエイムで3発命中→Victory」
  を確認済み

### 向き(構え)の仕様(2026-07-17)

- **通常(非構え)は移動方向を向く**。エイム入力がある間だけその方向を向き、
  向きを固定したまま移動できる(ストレイフ=置きエイムの手触り)
- エイム入力の出どころは入力デバイスごとに Godot 層が決める:
  - マウス: **右クリック保持中**、カーソル方向を AimDir として送る(右クリックに「構え」の仕事を与える)
  - ゲームパッド: 右スティックが倒れている間、その方向を送る(倒していなければ零=移動方向を向く)。
    標準的なツインスティック操作にそのまま溶ける
- **非構えでも射撃可(腰だめ)**。弾は向いている方向(=移動方向)へ出る。
  将来、腰だめは弾のブレ or 威力減のペナルティを入れて構えの価値を作る想定(現段階では未実装)
- ロジック層は「AimDir が零なら移動方向を向く(移動もなければ維持)」とだけ定義し、
  いつ AimDir を送るかはデバイス依存として Godot 層の責務にする
- スキル発動ボタンは右クリックから **E / Q** へ移す予定(スキル実装時)

### 左クリックの役割: クイックショット(2026-07-17)

レティクルが見えているのに非構えの射撃が移動方向へ飛ぶ不整合を解消。
左クリックを「単なる攻撃ボタン」から「速いが粗い射撃」へ:

- **非構え左クリック** = カーソル方向へのクイックショット(撃つ間はカーソルを向く)。
  将来スプレッド/威力減のペナルティを背負う側
- **右クリック構え** = 精密射撃(ストレイフ+ズーム+吸着)。狙う時間への投資
- クリック移動・カメラ操作への割り当ては不採用(WASD+ツインスティックの骨格と衝突する)
- Godot 層の入力変換のみの変更(構え中または射撃中に AimDir を送る)。
  ゲームパッドは元々右スティック=照準なので影響なし

### 当たる感の検証: 射撃場化(2026-07-17)

見下ろし+ツインスティックで「敵に当てるのが難しい」問題を、3D 化ではなく
**エイムアシスト+判定調整**で解決できるか検証する。そのためテストシーンを
ボス戦から**射撃場**に変更した:

- 敵(FSM・攻撃・勝敗)を撤去し、**動かない的**に置き換え。撃破すると一定 tick 後に
  乱数位置(壁マージン・プレイヤー至近を回避、seed 由来で決定論)にリスポーンする
- **エイムアシスト**: 発射方向と的の中心の角度差が吸着角(config、既定 10 度)以内なら
  弾を的の中心へ吸わせる。リスポーン待ち中は補正なし
- **判定調整**: 弾半径 2 → 3
- **命中統計**を検証指標として公開: ShotCount / HitCount / KillCount(State と HUD の acc%)。
  エージェントが「アシストあり/なしで命中率がどう変わるか」を数値で比較できる

敵の攻撃パターン(FSM)は検証が済んだら改めて再設計して戻す。

### エイム感の検証: カメラとヒット演出(2026-07-17)

射撃場での所感「エイムしてる感がない」への対策。原因は見下ろしそのものではなく
「部屋全体が常に見えていて、カメラが仕事をしていない」ことと診断し、次を導入:

- **部屋を画面より広く**(320×180 の窓で 640×360 の部屋を見る)。カメラという概念を作る
- **エイム連動カメラ**: カメラはプレイヤーとカーソルの内分点(非構え 25% / 構え 45%)。
  エイムした方向へ視界が伸び、「どこを見るか」の意思決定を生む。
  構え(右クリック)中はさらに 1.15 倍ズームイン(覗き込みの 2D 翻訳)
- **ヒットフィードバック**: 命中位置に広がるリング(ヒットマーカー)、的の白フラッシュ、
  2〜6 フレームのヒットストップ(撃破時は長め)
- 実装の骨格として、ロジックが **その tick の出来事(BattleEvent: BulletFired /
  TargetHit / TargetKilled)を公開**し、Godot 層が音・演出に翻訳する方式へ切り替えた。
  効果音・演出の発火を状態の差分から推測させない(弾 ID 差分方式は廃止)。
  カメラ・ヒットストップ・マーカーはすべて表現なので Godot 層に置き、ロジックは関知しない
