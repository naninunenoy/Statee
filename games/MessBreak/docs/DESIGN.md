# MessBreak 企画・設計

MessBreak 固有の企画・設計・開発記録はこの `games/MessBreak/docs/` に置く。
キャラ・敵・スキル・アイテム・ギミックのコンテンツ企画案は [SYSTEMS.md](SYSTEMS.md)。
Statee(フレームワーク)側の `docs/adr/` には持ち込まない。
Statee の ADR に書くのは、`src/` や repo 構成に波及する決定だけ。

## コンセプト

- 2D 見下ろし・リアルタイム戦闘のダンジョンアクション
- **ツインスティック型(移動とエイムの独立)+スキル主体、射撃は繋ぎ**。
  移動軸(WASD)と照準軸(マウス/右スティック)を分離し、
  ドッジ中もエイムが維持されることをコアにする。
  倒す主役はクールダウン制のスキルとその組み合わせで、通常射撃は低ダメージの繋ぎ
- 敵は物量(バンサバ型)ではなく、少数の敵と準備・配置の設計で面白さを作る
- **一言でいうと: 防衛線を築きつつ、キャラを切り替えて自ら前線を張る見下ろしシューター**
  (縦切り3の方向決め。下記)
- 内部は完全 2D 座標。見た目の奥行きは Godot の Y ソートで表現する(疑似 2.5D)
- キャラクター3人を切り替えて戦う。3人とも 18 歳の美少女:
  - アタッカー
  - バッファー(ヒーラー)
  - デバッファー

  キャラ切り替え=スキルセット切り替え。切り替えを挟むスキルコンボ
  (例: デバッファーでデバフ → アタッカーの大技を重ねる)が爽快感の中心

## 名前

- タイトルは **MessBreak**(表記はキャメルケースで統一)。
  「Mess(混沌・敵の群れ)を Break(打ち破る)」
- スラング上の問題・既存タイトルとの衝突がないことは確認済み(2026-07)。
  近縁名として Steam の「MESS」「Mecha BREAK」が存在する点のみ留意

## 開発方針

- 境界の掟(docs/USING.md)に従い、ルール・移動・当たり判定・敵 AI は
  `MessBreak.Logic`(純C#)に置き、xUnit + Shouldly でテストする。
  Godot 層は描画・入力・Statee 配線のみ
- 物理・衝突は Godot 物理に頼らず、Logic 層の自前判定(円・矩形)で行う
- 乱数は seed 注入、進行は tick 駆動で決定的に保つ
- ゲームバランスに関わる数値(速度・HP・攻撃力・部屋サイズ等)は
  ハードコードせず Logic 層の設定オブジェクト(BattleConfig / CharacterConfig)に集約する

## ゲームの枠(縦切り3の方向決め、2026-07-18)

「こういうゲームね」の一言を作るための方向決め。参照した先達は
陣取り+設置系(部屋を制圧すると設置権を得る構造)と、ボス戦前の準備が勝敗を
決める文化(設置物は守るためでなく**強敵を倒すため**の攻めの道具)。
プレイヤーの好みは「準備・編成・組み合わせの設計が答え合わせで報われる」タイプの
面白さであり、リアルタイムの操作精度をゲームの中心に置かない。
キャラ切り替え+スキルコンボは「仕込んだシナジーを現場で発火させるボタン」として核に残す。

## 現在の仕様

### ミッションループ(縦切り3-1、2026-07-18)

「小部屋1つ制圧 → 雑魚を倒して設置スロット解放 → タレットを置く →
出現ポイントをアトラクトして強敵出現 → 射撃+スキルコンボ+タレットで撃破」:

- 敵は **Enemy エンティティ(Mob / Boss)**。エイムアシスト・デバフ・スキル範囲は
  敵単位で効く(デバフは敵ごとに独立保持)
- **制圧**: 雑魚(HP 3 = アタッカースキル一撃)を倒すと ZoneCaptured、スロット解放
- **設置**: タレットは射程 160 内の最寄りの敵をクールダウン 30 tick ごとに自動射撃。
  タレット弾(Bullet.FromTurret)はプレイヤーの命中統計に混ぜない(デバフ倍率は乗る)
- **アトラクト**: 強敵(HP 20)が出現し、プレイヤーを追跡(接触距離で停止)。一度きり。
  強敵の攻撃は未実装(ループの成立検証に絞った)。プレイヤー HP は最小実装のみ
  (PlayerMaxHp=10、減らす手段が無いので常に満タン。UI 表示のために先行導入)
- **強敵撃破で MissionCleared**。HUD に進行ガイドを表示
- **Interactable 抽象**: 設置・アトラクトのような「場所に紐づく操作」は
  if 分岐でなく Interactable(Pos / Range / CanInteract / Interact)に一般化。
  入力は単一の `Interact`(F)で、**範囲内で発動可能な最寄りの1つだけが反応**する。
  宝箱・レバー・回復装置など今後のオブジェクトは実装1個の追加で乗る

### キャラ切り替えとスキル

- スキルは **E キー**で発動。**爆心はレティクル(カーソル)位置**。
  射程を超える指定はその方向の射程上限にクランプ。照準点を持たないデバイス
  (ゲームパッド)は AimPoint が null になり、向いている方向の射程いっぱいへフォールバック。
  発動はドッジ中不可・移動と両立。スキル命中は命中統計に数えない(KillCount は共通)
- 切り替えは**直接指定**: **1 キー=アタッカー、2 キー=デバッファー**(トグルではない。
  3人目は 3 キーで拡張)。切り替えクールダウン 30 tick。操作中キャラの再指定は不発扱い
- **スキルクールダウンはキャラごとに独立**(切り替えの存在意義)
- バランス値はキャラ単位の `CharacterConfig`(スキル CD・射程・半径・効果量)。
  デバフ倍率は付与時点のデバッファー設定値を保持(付与後に切り替えても変わらない)
- コンボの物差し: デバフ(被ダメージ2倍)→切り替え→大技(3)= 6。
  雑魚(HP 3)は大技単発、強敵は要コンボで削る
- デバフは足止めではなくダメージ増幅(足止め系は敵の攻撃パターン実装後に検討)

### 向き(構え)と射撃

- **通常(非構え)は移動方向を向く**。エイム入力がある間だけその方向を向き、
  向きを固定したまま移動できる(ストレイフ=置きエイムの手触り)
- エイム入力の出どころは入力デバイスごとに Godot 層が決める:
  - マウス: 構え(右クリック)中・射撃中・その余韻の間、カーソル方向を送る
  - ゲームパッド: 右スティックが倒れている間、その方向を送る
- **非構え左クリック** = カーソル方向へのクイックショット(速いが粗い射撃)。
  **右クリック構え** = 精密射撃(ストレイフ+ズーム)。
  将来、腰だめ側にスプレッド/威力減のペナルティを入れて構えの価値を作る想定(未実装)
- **エイムアシスト**: 発射方向との角度差が吸着角(既定 10 度)以内の敵の中心へ弾を吸わせる
- **命中統計**を検証指標として公開: ShotCount / HitCount / KillCount(State と HUD の acc%)

### カメラとヒット演出

- 部屋(640×360)は画面(320×180 の窓)より広い。カメラはプレイヤーとカーソルの内分点で、
  エイムした方向へ視界が伸びる。構え中は 1.15 倍ズームイン(覗き込みの 2D 翻訳)
- ヒットフィードバック: ヒットマーカー、敵の白フラッシュ、2〜6 フレームのヒットストップ
- ロジックが **その tick の出来事(BattleEvent)を公開**し、Godot 層が音・演出に翻訳する。
  効果音・演出の発火を状態の差分から推測させない。
  カメラ・ヒットストップ・マーカーは表現なので Godot 層に置き、ロジックは関知しない

### UI(2026-07-18)

- ストレッチ不使用。UI は実ピクセルで一定サイズ、ゲームはウィンドウが広いほど視界が広がる
  (絵の倍率は Zoom 3 固定。視界が部屋より広い軸はカメラを部屋中心に固定。最小 640x360)
- **下部 96px の UI バー**(ゲーム画面の外): 左=プレイヤー HP(数値+バー)、
  中央=キャラ2枠(アクティブ表示+スキル CD)+切替 CD
- **ミッションガイド**はゲーム領域の左上に縁取り付きで重ねる
- **Esc ポーズメニュー**: 再開 / はじめから(同 seed で作り直し) /
  中断(タイトル画面実装まで無効) / ゲーム終了。ポーズ中は論理 tick と演出タイマーを停止し、
  State に Paused を公開(Statee の freeze とは独立)
- 撃破数・命中率などの検証値は画面に出さず State(game/messbreak)で見る
- **UI の見た目は `ui/hud` State で公開**: ロジック値の再掲ではなく、実際に画面へ出している
  Label の文字列・HP バーの塗り率・各要素の画面上 Rect("x,y,w,h")・ポーズメニューの表示有無を
  ノードから写す。「ロジック→UI の写し間違い」「配置の破綻」をエージェントが検証するための入口。
  Rect はレイアウト確定前(起動直後 tick 0)は仮値になることがあり、1 tick 進めてから読む

### 操作まとめ

| 操作 | キー | tick トークン |
|---|---|---|
| 移動 | WASD / 矢印 | left/right/up/down |
| 射撃(クイックショット) | 左クリック / Z / J | fire(+aimx/aimy) |
| 構え | 右クリック保持 | (aimx/aimy を送る) |
| ドッジ | Space(移動入力必須) | dodge |
| スプリント | 左 Shift | sprint |
| スキル | E | skill(+skillx/skilly) |
| キャラ切り替え | 1 / 2 | char1 / char2 |
| インタラクト(設置・アトラクト) | F | interact |

## 縦切りの歩み(要約)

- **縦切り1(2026-07-16)**: 部屋1つ+敵1種(FSM)+近接攻撃+勝敗判定。
  その後のツインスティック化で近接と敵 FSM は廃止。
  **敵の攻撃パターンは「当たる感」検証が済んだ現在、再設計して戻す予定**
- **縦切り2(2026-07-16〜17)**: ツインスティック化(エイム分離+射撃)→ スキル1個
  (クールダウン+範囲ダメージ)→ キャラ切り替え+デバフコンボ。
  途中、「当てるのが難しい」対策としてテストシーンを射撃場(リスポーンする的)にし、
  エイムアシスト・判定調整・命中統計・カメラ・ヒット演出を導入
  (的は縦切り3-1 で敵エンティティに置き換えて廃止)
- **縦切り3-1(2026-07-18)**: 制圧→設置→アトラクト→強敵撃破の最小ループ。
  headless E2E でフルループ(タレット弾へのデバフ倍率含む)を確認済み

次の候補: 強敵の攻撃+プレイヤー HP(緊張の導入)、部屋の複数化(前進のリズム)、
設置物のバリエーション(壁・デコイ等、準備の選択肢)。
