## メインスレッドディスパッチ

- コマンドハンドラはソケットスレッドで走る(D-018)。Godot API を触るコマンド
  (drop 等)はメインスレッドに運ぶ必要がある
- `CallDeferred` は投げっぱなしで結果が返せない。コマンドは応答を返す必要があるので、
  「キューに積む → メインループが Pump → 完了までソケットスレッドをブロック」の
  同期ディスパッチにする
- メインスレッド自身から Run を呼ぶとデッドロックする。フレームワークでは検出せず
  「ソケットスレッド専用」と割り切る(タイムアウトが保険)

## Godot の SynchronizationContext に足をすくわれた(実バグ)

- 症状: PingTarget で quit / mainthread コマンドが必ず5秒タイムアウト。
  一方 ping / state は正常、フレームも進行 → Pump は毎フレーム呼ばれているはずなのに謎
- 原因: Godot .NET は await の継続をメインスレッドへ戻す SynchronizationContext を
  インストールする。StateeTcpServer の await に ConfigureAwait(false) が無かったため、
  「ソケットスレッドで動くはずのリクエスト処理」が実は**メインスレッド上**で動いていた。
  そこで MainThreadDispatcher.Run がブロック → Pump を呼ぶべきスレッド自身が待つ
  自己デッドロック → タイムアウト
- 教訓: Godot に埋め込むサーバー/バックグラウンド処理の await には必ず
  ConfigureAwait(false)。「どのスレッドで動いているか」は Godot では
  OS.GetThreadCallerId() / GetMainThreadId() で実測できる(mainthread コマンド)
- ping / state が「正常に見えた」のも実はメインスレッド実行だった
  (ブロックしないから露見しなかっただけ)。スレッド前提の検証は
  ブロック系コマンドを流さないと気付けない

## スイカゲーム最小シーン(物理 ↔ ロジックの境界)

- **接触の重複報告**: RigidBody2D の contact monitor は衝突の両側から
  `body_entered` を発火する → `ReportContact(a,b)` と `ReportContact(b,a)` が両方届く。
  ロジック側が「1回目で Entity を消す → 2回目は未知 ID として無視」で冪等なので
  Godot 層でのデデュープは不要。境界の契約として「未知 ID は無視」が効いている
- **溢れ判定は Area2D をやめて毎フレームの位置走査にした**。Area2D 案の問題:
  (1) 投下位置がライン より上なので、落下中のフルーツが必ずゾーンを通過して誤検知する
  (2) 大きいフルーツは投下時点でゾーンに重なる。
  → `_PhysicsProcess` で「一度でも何かに接触した(HasContacted)フルーツの上端が
  ラインより上か」を全走査して `SetOverflowing`。落下中の除外は接触フラグで表現でき、
  猶予時間(D-024 の 1.0s)はロジック側が持つので Godot 層にタイマーは不要
- **物理コールバック中のボディ操作**: `body_entered` は物理フラッシュ中に呼ばれるため、
  合体で消す方は `QueueFree`(安全)、生成する方は `Callable.From(...).CallDeferred()`
  で次フレームに遅延。合体イベント(Merges)が同期発火するのはこのため扱いに注意
- **合体後の生成位置はイベントに載っていない**(MergeEvent は ID と種類だけ)。
  位置は Godot 層が所有する(D-024)ので、消える2体の中点を Godot 層で計算する。
  ボディを消す前に位置を読む順序依存がある
- **headless スモークの学び**: チェリー2個を左右 80px 離して落とすと、
  半径 16+16=32 で着地後に届かず永遠に合体しない → 縦積みに変更。
  「物理任せの検証は初期配置が結果を決める」。終了条件は固定時間でなく
  Merges の発火(状態シグナル)で待つ形にした
