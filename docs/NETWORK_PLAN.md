# ネットワーク対戦ゲームへの展開計画(構想段階)

FPS/TPS + ネットワーク対戦(8人程度)+ NPC・破壊/獲得オブジェクト多数、という
ゲームをこのリポジトリで作ると仮定したとき、Statee に必要になる機能・ライブラリ・
新規開発内容を整理したもの。**FPS 級の内容は現時点では構想であり、着手は未決定**。

基盤の第一歩(リバーシのネット対戦化)は**完了済み**: 題材はリバーシ
(ローカル2人対戦完成後にネット化。docs/REVERSI_ROADMAP.md)、方針は **D-050**
(サーバ権威 = 純C#コンソール + コマンドレプリケーション + `syncee/` 別出し +
ITransport 抽象(フェイク / LiteNetLib)+ 自サーバー形態固定)と **D-051**
(マルチインスタンス検証はシナリオ語彙の拡張のみ)、**D-052**(合言葉によるマッチング
ゲート)。進め方・完了内容は docs/REVERSI_NETWORK_ROADMAP.md(N-0〜N-7)を参照。
以下の記述のうち D-050/D-051/D-052 と重なる部分はそちらが正。

## 現状の到達点(リバーシで実証済み)

- **サーバ権威 + コマンドレプリケーション**: `syncee/`(`ITransport`/`AuthorityLog`/
  `ReplicaLog`)、`game/Reversi.Server`(純C#コンソール権威サーバ)、
  `game/Reversi.Godot`(クライアント化)。ワイヤは MemoryPack、TOON は観測用
- **実トランスポート**: `Syncee.LiteNetLib`。テストの主戦場はフェイクトランスポート
  (`Syncee.Fake`)、実ソケットの E2E は薄く保つ(テストピラミッド)
- **切断検知**: 対局中の切断は相手の不戦勝として扱う。再接続は未対応
- **マルチインスタンス検証語彙**: `target(name, port:)` / `on(name) { ... }` /
  `wait_all([name,...], ...)` を Statee.Scenario に追加(D-051)。起動オーケストレータは
  作らず、`tools/run-reversi-network.ps1` に手順を置くだけに留めた(サーバ1+クライアント2の規模)
- **合言葉によるマッチング**(D-052): LiteNetLib の `connectionKey` を流用した
  最小ゲート。真のマッチメイキング(ランダムマッチ・複数ルーム同時捌き・認証)は未着手
- **明示的にやらなかったこと**(会話で確認済み。将来必要になれば設計する):
  - **状態スナップショット配布**: 確定コマンドの追記のみで、途中参加・観戦・再接続時の
    状態復元ができない。これが「観戦機能が無い」「途中参加できない」の直接の原因
  - **座席とクライアントの紐付けの強制**: 座席(黒/白)は接続順で決まるが、
    どのクライアントがどの色のコマンドを送れるかの制限は無い(なりすまし対策なし)
  - **本格的なマッチメイキング**: ロビー一覧・ランク・複数ルームの同時運用は
    D-052 のスコープ外(1サーバプロセス = 1合言葉 = 1対局)
  - **ホスト権威型 P2P / EOS のような外部バックエンド**: ホスティング形態は
    自サーバー固定のまま。将来の移植性は「確定コマンドの順序付きログ」という
    プロトコル設計の形態非依存性で担保する方針(D-050)であり、実装はしていない

## 抽象化の指針(ITransport 設計から得られた教訓)

ネットワーク抽象の形は最初から一般化して決め打ちにせず、**「ドメインを持ち込まない」
という境界だけを先に固定し、インターフェースの粒度はユースケースが要求する範囲でしか
広げない**という進め方をした。

- `ITransport` は「フェイク」「LiteNetLib」という異なる性質の2実装を最初から
  想定してから切った(1実装しかない段階で抽象化すると歪む)
- 最も厳しい制約(フェイクトランスポートでの freeze/step/切断注入の自由度)に
  合わせて最小形(送受信+切断)に留め、後から必要が証明されてから広げる
  (今回の合言葉機能も、既存の `connectionKey` を転用しただけで新規の抽象は増やしていない)
- リアルタイム・高頻度・大人数(FPS 級)になれば、信頼/非信頼チャネルの使い分けや
  優先度付き送信などで今の最小形では足りなくなる可能性が高い。その必要が
  具体的に証明されるまでは先回りして広げない

## 結論(要約)

Statee の基本コンセプト(State / Command / Log + 決定論 + wait)はそのまま通用する。
ただし現状は「単一プロセス・単一プレイヤー・低頻度スナップショット」前提であり、
足りないのは大きく3系統:

1. **複数インスタンスのオーケストレーション**(サーバ1 + クライアントN)
2. **大量エンティティ向けの State 設計**(フィルタ / 差分 / 集計)
3. **リアルタイム・非決定な世界の検証語彙**(許容付き assert、ローカル検証モード)

ネットワーク同期ライブラリ自体は新規導入が必要。

## ゲーム側アーキテクチャの推奨

**サーバ権威 + クライアントはコマンド送信・補間表示**。

- ARCHITECTURE.md の「コマンド駆動はネット対戦を意識」がそのまま活きる。
  **サーバへのコマンド注入 = Statee のコマンド注入と同型**にできるのが最大の資産
- 「ロジックは純C#・Godot 層は薄く」(HANDOVER 原則2)に従い、
  同期層も純C#に置く。Godot High-Level Multiplayer に乗るとロジックが
  Godot 依存になり原則と衝突する

## 導入候補ライブラリ

| 領域 | 候補 | 備考 |
|---|---|---|
| ネットワーク同期 | [LiteNetLib](https://github.com/RevenantX/LiteNetLib) + 自前同期層 | 純C#。Godot HLM(ENet)は原則2と衝突するため非推奨 |
| ワイヤシリアライズ | [MemoryPack](https://github.com/Cysharp/MemoryPack) | Cysharp 系で既存スタックと相性良。TOON は人間/AI 向け観測用、MemoryPack はワイヤ用と役割分担 |
| 3D 物理 | Godot 物理を継続(D-011 踏襲)。決定論が必要になったら BepuPhysics 2 | サーバ権威ならサーバ側でも物理が要る点に注意 |
| ナビゲーション(NPC) | Godot NavigationServer、純C#なら [DotRecast](https://github.com/ikpil/DotRecast) | NPC の意思決定(behavior tree / state machine)は純C#で自作し厚くテストする |

## Statee に新規開発すべき機能

### 1. マルチインスタンス・オーケストレーション(最重要)

8人対戦の検証は「サーバ1 + クライアントN」を起動し、それぞれに Statee で接続する形になる。

- **セッション管理・宛先指定** ✅: 複数ターゲット(port 違い)を1つのシナリオから扱う
  `target(name, port:)` / `on(name) { ... }` を実装済み(D-051。リバーシで実証)
- **クロスインスタンスの wait** ✅: `wait_all([name,...], path, field, op, value)` を
  実装済み(D-051)。「client1 で撃った弾のヒットが server と client2 の両方の State に
  現れるまで待つ」に相当する用途をリバーシの盤面一致検証で実証した
- **起動オーケストレータ**(未着手): サーバ + N クライアントの起動・シード/設定注入・終了を
  面倒みるランチャー(Statee.Cli 拡張 or 新プロジェクト。仮称 Statee.Fleet)。
  リバーシ(サーバ1+クライアント2)は `tools/run-reversi-network.ps1` という
  手順メモで足りたため、本格的なオーケストレータはまだ要らない(D-051 の判断)
- 8インスタンス級が必要になったら: PingTarget をネット対応させた最小ダミーで
  まず headless クライアントを8個立てられるかを検証する(D-013 の流儀)

### 2. 大量エンティティ向けの State スキーマ

NPC・破壊可能オブジェクトが数百になると、全量スナップショットの pull は重く
AI のコンテキストも溢れる。

- **State クエリ/フィルタ**: `state game/entities --filter type=npc`、
  `--near player1 --radius 10` のような絞り込み。全量ダンプは AI に読ませない
- **差分 State**: 前回スナップショットからの diff 取得(TOON のまま「変わったものだけ」)
- **集計 State**: 「NPC 残数」「破壊済みオブジェクト数」「チーム別スコア」のような
  AI が照合しやすい集約ビューを一級化。D-032 の Publishes / Explain の流れで
  State の自己記述も
- 安定 ID(D-038)は既にあるので、エンティティにもそのまま適用する

### 3. リアルタイム・非決定な世界の検証語彙

freeze / step は単一プロセスでは有効だが、ネット対戦では**全インスタンス一斉 freeze が
不可能**(ソケットタイムアウト、サーバの tick が進む)。

- **決定論の再定義**: 「座標一致はしない、State/Log ベースで検証」(D-011)をさらに進め、
  **許容付き assert**(HP が 80±5、位置がゾーン A 内、イベント順序の部分一致)を
  シナリオ語彙かターゲット側コマンドにする
- **ローカル検証モード**: 純C#のサーバロジックを単体プロセスで動かし、ネットワークを
  インプロセスのフェイクトランスポートに差し替えて freeze / step / シード注入が全部効く
  モード。**これがテストの主戦場**。実ソケットのマルチインスタンス E2E は薄く
  (テストピラミッドの維持)
- **Log の強化**: 複数インスタンスのログを時刻 / tick 同期して突き合わせる
  マージビュー(D-034 レポートの拡張)

### 4. NPC・オブジェクトまわり

- NPC の意思決定を純C# + シード注入で決定論化し、「NPC に台本を与えるコマンド」
  (scripted NPC)を Statee コマンドとして用意する。対人検証の相手役を AI が制御できる
- 破壊 / 獲得は「HP・耐久・所有者」を Arch のコンポーネントで持ち、
  イベント(破壊した / 拾った)を必ず Log に流す。検証は座標でなくイベントで行う

コマンドレプリケーション + ITransport 抽象)自体は流用できる見込み。
