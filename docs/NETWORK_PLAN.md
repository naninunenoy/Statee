# ネットワーク対戦ゲームへの展開計画(構想段階)

FPS/TPS + ネットワーク対戦(8人程度)+ NPC・破壊/獲得オブジェクト多数、という
ゲームをこのリポジトリで作ると仮定したとき、Statee に必要になる機能・ライブラリ・
新規開発内容を整理したもの。**現時点では構想であり、着手は未決定**。
着手する際は、まずアーキテクチャ方針(サーバ権威 + フェイクトランスポート)を
D-xxx として ADR 化するところから始める。

## 結論(要約)

Statee の基本コンセプト(State / Command / Log + 決定論 + wait)はそのまま通用する。
ただし現状は「単一プロセス・単一プレイヤー・低頻度スナップショット」前提であり、
足りないのは大きく3系統:

1. **複数インスタンスのオーケストレーション**(サーバ1 + クライアントN)
2. **大量エンティティ向けの State 設計**(フィルタ / 差分 / 集計)
3. **リアルタイム・非決定な世界の検証語彙**(許容付き assert、ローカル検証モード)

ネットワーク同期ライブラリ自体は新規導入が必要。

## ゲーム側アーキテクチャの推奨

**サーバ権威 + クライアントはコマンド送信・補間表示**。

- ARCHITECTURE.md の「コマンド駆動はネット対戦を意識」がそのまま活きる。
  **サーバへのコマンド注入 = Statee のコマンド注入と同型**にできるのが最大の資産
- 「ロジックは純C#・Godot 層は薄く」(HANDOVER 原則2)に従い、
  同期層も純C#に置く。Godot High-Level Multiplayer に乗るとロジックが
  Godot 依存になり原則と衝突する

## 導入候補ライブラリ

| 領域 | 候補 | 備考 |
|---|---|---|
| ネットワーク同期 | [LiteNetLib](https://github.com/RevenantX/LiteNetLib) + 自前同期層 | 純C#。Godot HLM(ENet)は原則2と衝突するため非推奨 |
| ワイヤシリアライズ | [MemoryPack](https://github.com/Cysharp/MemoryPack) | Cysharp 系で既存スタックと相性良。TOON は人間/AI 向け観測用、MemoryPack はワイヤ用と役割分担 |
| 3D 物理 | Godot 物理を継続(D-011 踏襲)。決定論が必要になったら BepuPhysics 2 | サーバ権威ならサーバ側でも物理が要る点に注意 |
| ナビゲーション(NPC) | Godot NavigationServer、純C#なら [DotRecast](https://github.com/ikpil/DotRecast) | NPC の意思決定(behavior tree / state machine)は純C#で自作し厚くテストする |

## Statee に新規開発すべき機能

### 1. マルチインスタンス・オーケストレーション(最重要)

8人対戦の検証は「サーバ1 + クライアントN」を起動し、それぞれに Statee で接続する形になる。

- **セッション管理**: 複数ターゲット(port 違い)を1つのシナリオから扱う。
  Ruby シナリオ語彙に宛先指定(例: `on(:client2) { send ... }`)
- **クロスインスタンスの wait**: 「client1 で撃った弾のヒットが server と client2 の
  両方の State に現れるまで待つ」— 同期検証の中核
- **起動オーケストレータ**: サーバ + N クライアントの起動・シード/設定注入・終了を
  面倒みるランチャー(Statee.Cli 拡張 or 新プロジェクト。仮称 Statee.Fleet)
- 最初のスライス: headless クライアントを8個立てられるかの検証。
  PingTarget をネット対応させた最小ダミーで行う(D-013 の流儀)

### 2. 大量エンティティ向けの State スキーマ

NPC・破壊可能オブジェクトが数百になると、全量スナップショットの pull は重く
AI のコンテキストも溢れる。

- **State クエリ/フィルタ**: `state game/entities --filter type=npc`、
  `--near player1 --radius 10` のような絞り込み。全量ダンプは AI に読ませない
- **差分 State**: 前回スナップショットからの diff 取得(TOON のまま「変わったものだけ」)
- **集計 State**: 「NPC 残数」「破壊済みオブジェクト数」「チーム別スコア」のような
  AI が照合しやすい集約ビューを一級化。D-032 の Publishes / Explain の流れで
  State の自己記述も
- 安定 ID(D-038)は既にあるので、エンティティにもそのまま適用する

### 3. リアルタイム・非決定な世界の検証語彙

freeze / step は単一プロセスでは有効だが、ネット対戦では**全インスタンス一斉 freeze が
不可能**(ソケットタイムアウト、サーバの tick が進む)。

- **決定論の再定義**: 「座標一致はしない、State/Log ベースで検証」(D-011)をさらに進め、
  **許容付き assert**(HP が 80±5、位置がゾーン A 内、イベント順序の部分一致)を
  シナリオ語彙かターゲット側コマンドにする
- **ローカル検証モード**: 純C#のサーバロジックを単体プロセスで動かし、ネットワークを
  インプロセスのフェイクトランスポートに差し替えて freeze / step / シード注入が全部効く
  モード。**これがテストの主戦場**。実ソケットのマルチインスタンス E2E は薄く
  (テストピラミッドの維持)
- **Log の強化**: 複数インスタンスのログを時刻 / tick 同期して突き合わせる
  マージビュー(D-034 レポートの拡張)

### 4. NPC・オブジェクトまわり

- NPC の意思決定を純C# + シード注入で決定論化し、「NPC に台本を与えるコマンド」
  (scripted NPC)を Statee コマンドとして用意する。対人検証の相手役を AI が制御できる
- 破壊 / 獲得は「HP・耐久・所有者」を Arch のコンポーネントで持ち、
  イベント(破壊した / 拾った)を必ず Log に流す。検証は座標でなくイベントで行う

## 進め方

原則(D-013)どおり、FPS 本体より先に **「PingTarget のネット版」**
(サーバ1 + クライアント2、カウンタを同期するだけの最小ダミー)で
上記 1 と 3 を縦切りスライスする。順序案:

1. ADR: サーバ権威 + フェイクトランスポートの方針を D-xxx 化
2. フェイクトランスポート上でカウンタ同期(純C#・ユニットテスト)
3. LiteNetLib 実トランスポートで同スライス(サーバ1 + クライアント2 headless)
4. シナリオ語彙の宛先指定 + クロスインスタンス wait
5. 8インスタンス起動オーケストレータ + 負荷確認
