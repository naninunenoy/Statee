# D-075 接続先プロセスの同一性確認(system/identity)とポート衝突時の即死

## 決定

「CLI が意図しないプロセス(古いバイナリ・残存インスタンス)に接続していた」事故への
二段構えを標準にする。

1. **`system/identity` State を全ゲーム共通で公開する**。
   `IdentityStateProvider`(Statee.Core)を `StandardCommands.Register` が自動登録する。
   公開するのは Pid / StartedAt / AssemblyName / **Mvid(ビルドごとに一意)** /
   AssemblyPath / AssemblyBuiltAt(後2つはベストエフォート)。
   検証シナリオの冒頭で「繋がっている相手はいま起動したプロセス・いまビルドしたバイナリか」を
   機械的に確認できる
2. **ポートを確保できないターゲットは即終了する(exit 1)**。
   `Main.StateeServer.cs` の定型で `StateeTcpServer.Start()` の SocketException を捕まえ、
   エラーを出して `Quit(1)` する。/new-game の雛形にも反映

## 背景

- MessBreak の UI 開発中、見た目確認用のウィンドウ版が port 9310 を握ったまま残り、
  新しい headless はバインド失敗をログに吐いた後も**サーバー無しで動き続け**、
  CLI は旧バイナリに接続していた。「未知の State パス」という無関係に見えるエラーから
  逆算してようやく気づいた(docs/adr/notes/game-dev-feedback-messbreak-slice1.md)
- Godot mono はアセンブリをストリームロードするため `Assembly.Location` が空になる。
  ファイルパス・タイムスタンプに頼らず、MVID を同一性の主キーにした

## トレードオフ

- 即死はゲーム側(Main.StateeServer.cs)の定型に置いた。Statee.Remote 側で
  プロセス終了まで面倒を見るのは通信層の責務を超える(Godot 非依存でもある)ため、
  「throw する Remote+捕まえて Quit するゲーム定型」の分担とした。
  既存ゲームへの一括適用はせず、雛形と MessBreak から広げる
- ポート自動割当(port=0)への切り替えは見送り。CLI/MCP 側の接続先発見の仕組みが
  必要になり、固定ポート運用(D-018)の単純さを失う

(identity 取得・ポート衝突で即 exit 1)。
