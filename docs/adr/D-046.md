# D-046: ソリューションを分割する(Statee.slnx はフレームワークのみ、ゲームは各自の slnx)

## 決定

1. `Statee.slnx` は フレームワーク(`src/` + `declaree/` + そのテスト + `sandbox/`)のみとする
2. 各ゲームは専用の `game/<Name>.slnx` を持つ(Logic / Godot / Logic.Tests +
   デバッグ用ビューとして参照している `src/` プロジェクト)
3. `/new-game` skill はゲームの slnx を生成し、`Statee.slnx` には触れない
4. フレームワーク変更後の追従確認は `tools/build-all.ps1`(全 slnx を順に
   ビルド+テストする門番)で行う

## 背景

- D-013(フレームワーク開発にゲームの事情を持ち込まない)・D-045(ゲーム開発者向け
  導線の分離)の対称として、「ゲーム開発にフレームワーク全体の事情を持ち込まない」を
  ソリューション構造でも表す
- ゲーム側 slnx に `Statee.Mcp` が入らないため、「MCP サーバー実行中はソリューション
  ビルドが exe ロックで失敗する」問題がゲーム開発では構造的に消える
- ソリューションはビューであり、ビルドは ProjectReference を辿る。分割しても
  依存解決は変わらず、技術的リスクはない

## トレードオフ

- 「全部まとめてビルドする門番」が暗黙には無くなる。フレームワークの API 変更が
  ゲームを壊しても即座には気づけない。`tools/build-all.ps1` の実行を
  ルート CLAUDE.md の環境の知識に組み込んで運用で担保する(将来は CI へ)
- slnx が3つ以上に増える。ゲーム追加ごとに1つ増えるが、生成は skill が担う

## 状態

採用。
