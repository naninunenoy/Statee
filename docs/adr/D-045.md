# D-045: ゲーム開発者向け導線は USING.md + /new-game skill を正典とする(サンプルは非規範)

## 決定

1. このリポジトリで自作ゲームを作る人(と AI)向けの入口として `docs/USING.md` を置く。
   境界の掟・Statee 配線チェックリスト・動作確認の回し方を**自己完結**で記述する
2. 新しいゲームの雛形生成は `/new-game` skill(`.claude/skills/new-game/`)が担う。
   csproj / project.godot / Statee 配線済み Main.cs 等の**テンプレートを skill 内に焼き込み**、
   生成直後にビルド・テスト・headless 疎通が通る状態を保証する
3. サンプルゲーム(SuikaGame / RogueGame)は「動くデモ」であり**規範ではない**。
   文書・skill からサンプルの特定行を写経元として参照しない。
   サンプルは今後変更・削除されうる
4. `/verify` skill はターゲット決め打ちをやめ、`--path <dir>` 引数で任意のゲームに使う

## 背景

- リポジトリの利用者は「フレームワークを開発する人」と「これを使ってゲームを作る人」の
  2種で、大半は後者になる想定。しかし既存文書(HANDOVER / ARCHITECTURE / ADR)は
  すべて前者向けだった
- RogueGame をゼロから組み上げた経験で、後者が詰まる箇所が具体化した:
  ボイラープレート(CopyLocalLockFileAssemblies、Generator の Analyzer 参照)、
  Statee 配線のイディオム(volatile スナップショット、RegisterMainThreadCommand、
  ping の自己登録)、動作確認の作法(import クラッシュ無視、カンマ区切り引数)
- 当初「最小の雛形+サンプル該当行へのリンク」を検討したが、サンプルは開発の実験台で
  あり変更されうるため、参照が腐る。テンプレートを skill に持たせて正典を1箇所にする

## トレードオフ

- テンプレートが skill 内に複製されるため、フレームワークの API 変更時は
  skill の追従が必要になる。サンプルのビルドが通っても skill のテンプレートが
  古びていることは検出できない(将来: skill を実行して生成物をビルドする検査も可)
- USING.md とルート CLAUDE.md・GUIDELINE.md に一部重複が生じる。
  USING.md は「読む順番の逆転(利用者視点)」が目的であり、詳細は各文書へ委譲して
  重複を最小に保つ
