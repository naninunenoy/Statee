# D-036: SuikaGame の UI を Declaree に移行し、game/ui State を ui/tree で置き換える

## 決定

1. SuikaGame の UI(タイトル画面・スコア HUD)を Declaree(D-035)で宣言する。
   UI ツリーは `(GamePhase, Score) → UiNode` の純関数 `BuildUi` で導出し、
   フェーズ・スコアの R3 購読から全再構築する(フェーズ別ツリー。Visible 切り替えは廃止)
2. 手書きの `UiState`(`game/ui`)・`_uiElements`・`Bind`/`Describe`/`UpdateUiState` を削除し、
   UI の State 公開は `ui/tree`(UiSnapshot による Rect 込み記述子、Declaree.Statee)に一本化する
3. D-032 の意味論は Declaree の語彙で引き継ぐ:
   - Publishes 相当 → `Button.OnClick` の ID を発行される VitalRouter コマンド型名
     (`nameof(StartGameCommand)` 等)にし、`Dispatch` がその型を発行する
   - Explain → UiNode 共通プロパティ `Explain` として語彙に追加(記述子の `explain`)
4. Godot 都合の配線はホスト(Main)の責任とする:
   - 描画ルートはツリー投入後に FullRect アンカーを設定
   - 全面に広がるルートは `MouseFilter = Pass` にして、プレイ中のクリック
     (`_UnhandledInput` → Drop)を飲み込まない
   - headless のビューポート 64x64 対策で `GetWindow().Size` を明示

## 背景

- Declaree の最初のスライスは PingTarget で E2E 済み(D-035)。実用ゲームでの検証が次段
- `game/ui` と Declaree の `ui/tree` は役割が重複する。両方維持すると
  「UI の実装と State 公開の乖離」という D-031/D-032 が避けた問題が再発する
- `game/ui` への依存はドキュメントと記録テストの文字列のみで、削除の障害はなかった
  (後方互換は求められない限り維持しない方針)

## トレードオフ

- 旧 `game/ui` の `Id`(ノード名の安定 ID)と `Interactable` は失われた。
  ツリー構造+`onClick`/`disabled`/`visible` で代替できるが、
  要素をパスで指名する語彙が必要になったら Declaree 側に Id を足す
- スコア更新のたびに全再構築する。UI が小さいため問題ないが、
  肥大化したら差分適用(D-035 で保留)の判断材料になる
- `ViewportWidth/Height` の明示フィールドは無くなった(ルートの Rect が同じ情報を持つ)

## 状態

採用。headless E2E 確認済み(タイトルツリー観測 → Rect からはじめるをクリック →
Playing 遷移 → スコア HUD(左上)→ drop / クリック投下のスコア反映 → quit exit 0)。
