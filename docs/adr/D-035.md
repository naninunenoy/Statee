# D-035: 宣言的 UI フレームワーク Declaree — IR 中心設計・C# DSL・Statee から独立

## 決定

1. **宣言的 UI フレームワーク「Declaree」を汎用フレームワークとして立ち上げる**。
   Statee ファミリーには入れない(namespace は `Declaree` / `Declaree.Godot`、
   `Statee.*` を使わない)。リポジトリ内の独立ディレクトリ `declaree/` に置く
2. **設計の中心は IR(中間表現)**: UI を Godot 非依存の C# record ツリー
   (`UiNode` 派生型)として表現する。UI 定義はこのツリーを返す純粋な C# 式
   (= C# が DSL)。Ruby など外部言語の DSL は採用しない
   (IR がシリアライズ可能なので、必要になれば後から外部フロントエンドを足せる)
3. **プロジェクト分割**:
   - `Declaree` … IR 定義 + reconciler(Godot 非依存の純C#。xUnit で厚くテスト)
   - `Declaree.Godot` … `UiNode` ツリー → Godot Control ノード変換(Godot 依存はここだけ)
4. **描画はまず全破棄・全再構築で始める**。ちらつき・フォーカス喪失・性能が
   問題になった時点で差分適用(reconciliation)を導入する
5. **イベントは ID ディスパッチ**: `UiNode` にイベント ID(文字列)を持たせ、
   Godot 側でシグナル受信時に ID を発行する。ハンドラのクロージャを IR に埋めない
   (IR のシリアライズ可能性を守る)
6. **Statee との接続は薄いアダプタのみ**: UiNode ツリーを State として公開する
   シリアライザを提供する。Declaree 本体は Statee に依存しない
   (依存方向: ゲーム → Declaree、アダプタ → 両方)

## 背景

- UI を宣言的に(状態 → UI ツリーの純関数として)記述したい。
  UI がデータになると「この状態のときこの UI」を純C#ユニットテストで検証でき、
  「Green means done」の対象に UI が乗る
- UI ツリーが構造化データなら Statee の `state` でそのまま公開でき、
  AI Agent が画面内容をスクショ解析なしで観測できる。既存の `game/ui` State
  (D-031)より根本的な解になり得る
- 当初 Ruby で DSL を書く案を検討したが、C# の record + 式で同等の書き味が得られ、
  型チェック・IDE 支援が効き、境界のシリアライズも不要になるため C# を選択。
  Ruby は「IR を吐く別フロントエンド」として将来の拡張点に残す
- Statee に入れない理由: Statee の責務は観測・駆動(観測する側)であり、
  UI ランタイムはゲームを作る道具(観測される側)。混ぜると両者の輪郭がぼやける
  (D-013 の精神)。最初から汎用フレームワークを狙うため `game/` 配下でもなく
  独立ディレクトリとする

## トレードオフ

- 全再構築方式は毎フレーム相当の再生成には耐えない。UI 状態の変化時のみ
  再構築する前提で始め、差分適用は必要が証明されてから(YAGNI)
- イベントを文字列 ID にするとハンドラ結線の型安全性は落ちる。
  ID 定数の一元管理か、将来のソースジェネレータで緩和する
- Godot の Control 体系(アンカー・サイズフラグ・テーマ)を IR の語彙に
  どこまで露出するかは実装しながら絞る。最初は最小語彙
  (VBox / HBox / Label / Button 程度)から始める
- リポジトリ同居のまま進めるが、他ゲーム・他エンジンから使いたくなった時点で
  独立リポジトリ / NuGet 化を検討する(分離の引き金)
