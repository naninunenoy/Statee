# D-031: 画面遷移は純C#の GameFlow、UI 導入は3スライスで進める

## 決定

タイトル画面・スコア表示の UI 導入を次の3スライスに分けて進める。

1. **シーン State と遷移の状態機械**(本スライスで実装済み)
   - 画面遷移の規則は純C#の `GameFlow`(SuikaGame.Logic)が持つ。
     フェーズは `GamePhase`(Title / Playing)、遷移は `StartGame()`(Title → Playing の一方向)のみ
   - Godot 層は `Phase` を購読して UI(タイトル画面 / スコア HUD)を差し替えるだけ
   - フェーズは `game/scene` State として外部へ公開し、`start` コマンドで遷移させる
   - `drop` はプレイ中のみ受け付ける。タイトルの「おわる」は既存の終了処理(quit と同じ経路)に合流する
2. **UI 幾何・操作可否の State 公開**(GUIDELINE §7-2。実装済み)
   - UI 要素の Id・テキスト・Rect・可視性・interactable を `game/ui` State として公開し、
     「スコアが UI に表示されている」を幾何述語で検証可能にする
   - 幾何述語の基準として ViewportWidth / Height も含める
3. **InputEvent 注入コマンド**(GUIDELINE §7-3。実装済み)
   - `click`(x, y)コマンドが Viewport.PushInput でマウス移動+左クリックを注入する。
     UI のヒットテストを通るため、非表示・無効なボタンへのクリックは正しく「効かない」
   - `start` コマンド(ロジック直呼び)は AI 向けの近道として残す。
     受け入れテストでは `game/ui` の Rect から座標を取り `click` で操作する

## 背景

- タイトル画面(開始 / 終了)とスコア表示の導入により、初めて「本体が今どの画面にいるか」を
  AI が観測する必要が生じた。State の粒度「画面・シーンの内容」(PLAN)の最初の実装
- GUIDELINE §7 が予告していた機能要件(UI 幾何の State、InputEvent 注入、安定 ID)を
  実際に要求する最初のユースケースでもある

## トレードオフ

- **headless ではビューポートが 64×64 になる**(窓が無い Godot headless の挙動。
  project.godot の窓サイズは反映されない)。幾何述語は絶対座標ではなく
  `game/ui` の ViewportWidth / Height を基準に書く
- **「おわる」ボタンのクリックは応答が返らない**。プロセスが即終了しソケットが切れるため、
  CLI は「応答が無いまま切断された」を返す。終了の確認はプロセスの exit code で行う
- `start` / `click` が併存する。`start` は AI 向けの近道、受け入れテストは `click` を使う
- UI の StateProvider はゲーム側の手書き(`SceneState` / `UiState`)から始める。
  Control ツリーを歩く汎用プロバイダの `src/` への追加は、2例目が出るまで行わない(D-013, YAGNI)
- リザルト画面(GameOver → Result)は現時点で作らない。`IsGameOver` は盤面凍結として
  既に観測可能であり、必要になったスライスで `GamePhase` を拡張する

## 状態

採用。全3スライス実装済み(GameFlow + game/scene + start / game/ui / click)。
headless E2E で以下を確認済み:
Title → start/click → Playing、タイトル中の drop 拒否、フェーズ遷移で UI 可視性が反転、
合体後に ScoreLabel が「スコア: 1」、非表示ボタンへのクリック無効、
「おわる」クリックで exit 0 の正常終了。
