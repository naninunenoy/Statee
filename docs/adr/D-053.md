# D-053 第4サンプルゲームは2人協力ボス戦(RaidBoss)

## 決定

Syncee(D-050)がまだ実証していない領域(常時・高頻度・複数エンティティのリアルタイム同期)を
潰すため、第4サンプルゲームとして **RaidBoss**(2人協力ボス戦)を作る。

- 規模は最小に絞る: **プレイヤー2人 + ボス1体**
- 操作はリアルタイム(移動・攻撃を随時入力。ターン制ではない)
- 同期方式は**決定論ロックステップ**(設計は D-054 に分離して記録)
- CPU 対戦・観戦・3人以上への拡張は作らない(将来候補として ARCHITECTURE.md に記録するのみ)

進め方はリバーシと同じ4段階(スケルトン → 失敗するテスト → 実装 → リファクタ)を
フェーズ分割する。まずローカル(ネット非依存)で決定論なボス戦ループを完成させ、
Syncee 配線は後段のフェーズで足す。

## 背景

- リバーシ(D-050〜D-052)は「ターン制・低頻度・状態依存の合法性判定」を実証した。
  Syncee のコア抽象(`ITransport`/`AuthorityLog`/`ReplicaLog`)が本当にドメイン非依存かは、
  性質の異なる同期パターンに当ててみないと確認できない
- 高頻度・常時系のテーマ候補(タイピング対戦・早押しクイズ・多人数じゃんけん)も検討したが、
  「複数エンティティの決定論シミュレーションを跨ぎ同期する」という、Syncee にとって
  最も学びの大きい未踏領域を狙うため、協力ボス戦を選んだ
- スケルトン一式は `/new-game` スキルの雛形をそのまま使用(GameLogic/GameState/Main の
  プレースホルダ)。この段階では通常の1プロセス・ネット非依存ゲームと変わらない

## トレードオフ

- モンハン的な題材は本来ならリアルタイム物理・当たり判定・敵AI・途中参加(状態スナップショット
  配布)まで踏み込む余地があるが、それらは一度に決定せず、まず最小規模(2人+ボス1体)で
  ロックステップの骨格を通すことを優先した。物理エンジン(Godot 物理)を使うか、当たり判定を
  ロジック層の純粋計算で済ませるかは、実装フェーズで別途決める
- 3人以上・観戦は明示的に見送った。台数が増えるほど「両者揃うまで待つ」ロックステップの
  待ち合わせ設計が複雑になるため、まず2人構成で決定論同期そのものを検証してから広げる
