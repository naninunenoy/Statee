# D-026 pause / step は TimeControl(エンジン非依存)+ 標準コマンドで提供

- **決定**: `Statee.Core.TimeControl` が pause / resume / step の状態機械を持つ。
  - ゲームループは非ポーズ時に毎フレーム `OnFrame()` を呼び、`IsPaused` を
    エンジンのポーズ機構(Godot なら `SceneTree.Paused`)へ写す
  - `Step(n)` は n フレーム消費後に自動で再ポーズ。`WaitForStep` により
    **step コマンドは進行完了後に応答を返す**(AI が「進め終わった盤面」を直後に観測できる)
  - `RegisterTimeControl(host)` 拡張で pause / resume / step を標準コマンドとして登録
- **背景**: D-003(時間制御・決定論)の実装。ロジック層は外部 tick 駆動(D-023)なので、
  ポーズの実体は「Godot 層が物理とTickを止めること」に帰着する
- **トレードオフ**:
  - (+) 純C#で網羅的にテスト可能(コマンド登録込みで13テスト)。Godot 層は反映のみ
  - (+) pause/resume/step はスレッド安全なのでメインスレッドディスパッチ不要。
    step のブロック待ちも D-025 の自己デッドロック経路を通らない
  - (−) step の「1フレーム」は物理フレーム(_PhysicsProcess)単位。描画フレームとはずれ得る
- **Godot 統合で得た知見**:
  - ポーズ中も Pump / State 更新を動かすため Main は `ProcessMode = Always`。
    ただし **ProcessMode は子に継承される**ため、フルーツ(RigidBody2D)を明示的に
    `Pausable` にしないと物理が止まらない
  - `IsPaused` → `SceneTree.Paused` の反映は `_Process` で行い、`OnFrame()` は
    「実際にツリーが非ポーズだったフレーム」だけ数える(反映遅延の1フレームを数えない)
- **状態**: ✅ 実装完了(4段階コミット)。E2E: pause 中の drop でフルーツが凍結、
  step 30 で Y=120→236.6、同シードの合体結果が別セッションと完全一致(決定論の傍証)
