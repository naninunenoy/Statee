# D-058 RaidBossをシューティング演出化(弾の発射・着弾を決定論ロジックに追加)

## 決定

`RaidBoss.Logic.GameLogic` のAttack行動を「即ダメージ」から「弾を発射し
`ProjectileTravelTicks`(2Tick)後にボスへ着弾してダメージを与える」方式へ変更する。

- Attack行動は`Projectile(OwnerIndex, TicksRemaining)`を発射する。既存の弾は毎Tick
  `TicksRemaining`を1減らし、0以下になった弾はボスへ`PlayerAttackDamage`のダメージを
  与えて消滅する
- 発射した弾は当Tick中は進めない(次Tickから減り始める)ため、着弾までちょうど
  `ProjectileTravelTicks`かかる。乱数を使わない決定論的なカウントダウンのみで表現し、
  Godotの物理エンジンは使わない(既存の「今後の候補」に記載していた方針を踏襲)
- 操作不能中(D-057)のプレイヤーは弾を発射できない(Idle扱いのまま)
- RaidBoss.Godotの`_Draw`に、中央のボス(HPに応じて赤みが変わる円)、画面下部に並んだ
  各プレイヤー(操作不能中はグレー表示)、プレイヤーからボスへ向けて飛ぶ弾(進捗は
  `TicksRemaining`から算出)を描画するシューティング風の見た目を追加した

## 背景

- ユーザー要望3点のうち「中央にボスオブジェクトを表示し、シューティングゲームのように
  玉を発射しボスのHPを削る」を実装するスライス。ロビー機能(D-056/D-057)が先に完了して
  いたため、その後に着手する順とした
- 弾に実座標・当たり判定を持たせるとGodot物理エンジンとの結合が必要になり、決定論
  (D-011/D-049)を壊すリスクがある。ARCHITECTURE.mdの「今後の候補」に元々書いていた
  「当面はRaidBoss.Logic内の純粋計算のみ」という方針を維持し、弾の位置は座標ではなく
  残りTick数のみで表現した(見た目の座標補間はGodot側の描画専用ロジックで行い、
  ゲームロジックには一切持ち込まない)

## トレードオフ

- 弾の「見た目の速さ」はTick数(`ProjectileTravelTicks=2`)でしか表現できず、
  Godot側の描画は単純な線形補間(`Lerp`)に留まる。放物線や複数の弾種などの演出は
  今後の候補
- Attackが即時ダメージでなくなったことで、既存テストの多くを「着弾を待つ」形に
  書き換える必要があった(意図した仕様変更に伴う正当な修正)
