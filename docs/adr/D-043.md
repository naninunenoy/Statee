# D-043: エージェント向け入力 API はアクション単位(デバイス非依存)

## 決定

1. エージェント向けの操作 API は**アクション単位を正**とし、入力デバイスが増えても
   (ゲームパッド等)この面は変えない
   - UI 操作は name / id 指定(D-038, D-041)。座標決め打ちはしない
   - キー・ボタン操作は `game/input` State(D-039)でアクションを発見し、
     発行されるコマンド(Publishes)の意味に基づいて操作する。
     「どの物理キー/ボタンか」はエージェントが気にしない
2. 低レベル生入力(座標指定クリック、任意キー押下、任意パッドボタン・スティック注入)は
   **モンキーテスト用途として残す・提供する**。ただし通常の動作確認シナリオでは使わない
   (使うのは「想定外入力に対する頑健性」を確認したいときだけ)
3. 長押し・連打・同時押し等の**入力ジェスチャ → コマンドへの変換はゲームの Godot(入力)層**で行う
   - 純C#ロジック層は変換済みのコマンド(アクション)だけを受ける(D-023, D-024 と同じ境界)
   - エージェントには変換後のアクションを公開する。長押しをエミュレートさせるのではなく
     「そのアクションを発行する」コマンドを提供する(生の押下/解放注入はモンキーテスト用)
4. デバイス追加時(例: ゲームパッド)は Godot 層のバインド表(D-039)にエントリを足し、
   `game/input` の導出元を広げるだけにする。Statee 本体(`src/`)に
   デバイス固有の機能は持たせない(D-013)

## 背景

- ゲームパッド対応を検討した際、「Statee に足すべき機能は何か」を整理した結果、
  コア(State / Command / Logs の配管)はデバイスを知らないままでよく、
  必要なのは機能ではなく規約だと判断した
- D-032(UI の作用はコマンド型との対応で公開)、D-038(name 指定クリック)、
  D-039(キーバインドの State 公開)がすでにこの方向を向いており、
  本決定はそれらを「デバイスが増えても不変の原則」として明文化するもの

## トレードオフ

- 生入力注入(モンキーテスト面)とアクション面の二面を維持するコストがかかる。
  ただし生入力面は PushInput の薄いラッパーであり、実入力経路を通す検証
  (非表示要素には効かない等。D-038)にも必要なので削れない
- 入力ジェスチャの判定(長押し閾値等)を Godot 層に置くと、その部分は
  ユニットテストでなく E2E でしか検証できない。判定ロジックが複雑化したら
  「InputEvent 列 → アクション」変換を純C#に切り出してテストする
- ジェスチャ判定で壁時計を使うと freeze / step(D-026, D-040)と整合しない。
  経過時間は `_Process` の delta 等、時間制御の影響下にあるものを使う

## 状態

採用。規約のみ(コード変更なし)。ゲームパッドを使うサンプルが必要になった時点で
この原則に従って実装する。
