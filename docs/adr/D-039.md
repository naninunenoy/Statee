# D-039: キーバインドの State 公開(game/input)

## 決定

1. キー入力の作用を D-032 と同じ原則(**Publishes は配線から導出する**)で外部公開する。
   Godot 層にキーバインド表 `KeyBinding(Key, ActiveIn, Publishes, Explain, Publish)` を置き、
   この表を単一の情報源として
   - `_UnhandledInput` のキー処理(該当フェーズのバインドを探してコマンド発行)
   - `game/input` State(Key / ActiveIn / Publishes / Explain の一覧)
   の両方を導出する
2. ESC の処理は GameFlow の直接呼び出しをやめ、`PauseGameCommand` /
   `ResumeGameCommand` の発行に変える(PauseGameCommand を新設し Router に配線)。
   トグルは「Playing では PauseGameCommand、Paused では ResumeGameCommand」という
   ActiveIn 付きの2バインドとして表現する
3. バインド表は起動後不変とし、State スナップショットは一度だけ構築する

## 背景

- 「プレイ中に ESC を押すとポーズする」は実装や README を読まないと予想できず、
  CLI / MCP から観測できる材料が無かった(UI ボタンの onClick / explain と非対称)。
  AI は `state --path game/input` → `send --command key --arg key=escape` で
  発見から操作まで完結できるようになる
- InputMap(Godot のアクション定義)は使っていない(キーが ESC 1個のため。
  リマップやゲームパッド対応が要件になったら InputMap 化を検討し、その際も
  game/input の導出元をアクション定義に差し替えるだけにする)

## トレードオフ

- マウス操作(移動 = 投下位置、左クリック = 投下)はバインド表に載っていない。
  キーと違いイベント型の照合が多様で表に一般化しにくい。必要になったら
  game/input に Mouse セクションを足す
- ActiveIn は1フェーズ1バインド。複数フェーズで有効なキーが出たら配列にする

## 状態

採用。ロジックは Router のユニットテスト(39件緑)、Godot 層は headless E2E で確認済み
(`game/input` に Escape×2 が Publishes / Explain 付きで載り、ESC トグルも従来どおり)。
