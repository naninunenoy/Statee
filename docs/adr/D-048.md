# D-048: 第3サンプルゲームは固定タイムステップの横スクロールシューティング

## 決定

1. 第3のサンプルゲームとして、**リアルタイム × 完全決定論**の横スクロール
   シューティング(仮称 ShootingGame)を `game/` 配下に追加する
2. 論理は固定タイムステップ(60Hz)の `Tick()` 駆動とし、運動・衝突は自前の数式で
   実装して **Godot 物理を使わない**。乱数はシード由来の1系統のみ
3. エンティティは Arch(ECS。D-004)、ゲーム内イベントは VitalRouter(D-005)を
   多対多の Publish/Subscribe で本格活用する。VitalRouter の interceptor による
   **イベントログの記録と State 公開**を Statee の新しい検証の柱にする
4. 入力は「フレームごとの入力状態(押されているキーの集合)」として論理へ渡し、
   継続入力の注入 API(D-043 の拡張)とフレーム精度リプレイを実証する
5. 描画は emoji + 同梱 Noto Color Emoji サブセット(D-044 と同方式)。
   アセットはフォント1ファイルのみ

ゲームの詳細仕様と実装計画は `game/SHOOTING_PLAN.md` に置く
(形になった時点で削除される作業ドキュメント)。

## 背景

- 検証特性の象限が「リアルタイム×非決定(SuikaGame)」「ターン制×決定論
  (RogueGame)」の2つで、「**リアルタイム×決定論**」が空いていた。
  リアルタイムゲームでも物理エンジンを使わなければ完全決定論にでき、
  RogueGame で確立したリプレイ検証をフレーム精度へ拡張できる
- D-004(Arch)・D-005(VitalRouter)で採用したライブラリが本来の規模で
  使われていなかった(VitalRouter は SuikaGame で switch 1個)。
  大量エンティティ+多対多イベントの STG はどちらの適用例としても自然
- 継続入力(押しっぱなし)の注入は既存の入力 API(D-043)に無い語彙で、
  フレームワークを成長させる実需要になる

## トレードオフ

- サンプルが3つになり追従コストが増える(D-044 と同じ論点)。ただし
  D-046 のソリューション分割と tools/build-all.ps1 で管理コストは抑えられている
- マリオ型横スクロールも候補だったが、プラットフォーマーは衝突解決と
  「ジャンプの手触り」の官能調整が本体で、5分ゲームの枠に対して重い。
  Godot 物理に頼ると決定論を失い、本ゲームの狙いと矛盾するため見送った
- 60Hz 固定 Tick は可変フレームレート環境で描画とズレうる。論理と描画の
  分離(補間なしの割り切り)で対処する
