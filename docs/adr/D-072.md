# D-072 リアルタイムゲームの tick コマンド定型を RegisterTickCommand として Statee.Core へ吸収

## 決定

freeze + 「入力を指定して N tick 進める」プレイ経路(D-048/D-049)の定型を、
`StateeHost` の拡張メソッド **`RegisterTickCommand<TInput>`**(Statee.Core)として提供する。

- ライブラリ側が持つ: frames 引数の読み取りとクランプ(既定上限 3600)、
  step ループ、`TimeControl.OnFrame()` の呼び出し、メインスレッドコマンドとしての登録
- ゲーム側が渡すのは 3 つだけ: `parseInput`(引数 → 入力型。呼び出しごとに 1 回)/
  `step`(入力で 1 tick 進める)/ `result`(応答ペイロード。描画反映やログもここで行える)

## 背景

- ShootingGame(D-048/D-049)の tick コマンドを MessBreak がほぼ丸写しした。
  ゲームごとに違うのは「入力トークン → 入力型の写像」だけで、残りは毎ゲーム同じだった
  (docs/adr/notes/game-dev-feedback-messbreak-slice1.md の項目 1)
- D-047 は「配線の定型は libs へ」と定めているが、この定型は TimeControl と CommandArgs
  だけに依存しエンジン非依存なので、libs/Statee.Godot ではなく Statee.Core に置いた
  (テストも xUnit で直接書ける)

## トレードオフ

- 入力は呼び出しごとに 1 回だけ解釈する(N tick の間は同じ入力)。tick ごとに異なる
  入力列を送る用途(リプレイ再生)は対象外で、必要になったら別コマンドとして設計する
- result 内で描画反映(RefreshView)を行う規約はゲーム側の自由に委ねた
  (強制するとゲームごとの後処理の差を吸収できない)

## 状態

採用。Statee.Core.Tests に仕様テスト 5 件緑。ShootingGame と MessBreak を移行済み。
