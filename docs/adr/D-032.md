# D-032: UI の作用は VitalRouter コマンド型との対応で公開する(配線からの導出のみ)

## 決定

1. **操作可能な UI 要素は「操作されたとき何を発行するか」を State として公開する**。
   `game/ui` の要素に `Publishes`(発行される VitalRouter コマンドの型名)と
   `Explain`(自然言語の説明)を載せる
2. **公開情報は配線そのものから導出する**。ボタンの配線は宣言的な `Bind(button, command, explain)`
   に一本化し、(a) 実際の Pressed → コマンド発行の配線と (b) State への公開が
   同じ一次情報から出るようにする。手書きの説明文と実装が乖離しうる形
   (配線はクロージャ、説明は別途文字列)を構造的に禁止する
3. **2層の役割分担**: `Publishes` は機械導出で嘘をつけず、AI が検証計画の根拠にしてよい。
   `Explain` は人間が書くヒントであり、理解を助けるが検証の根拠にはしない
4. **VitalRouter の初使用はこの UI 作用宣言に限定する**。
   既存の境界 — Godot → ロジックの直接メソッド呼び出し(ReportContact / SetOverflowing / Tick)、
   ロジック → Godot の R3 購読(Merges / Score / Phase)、Statee のコマンドハンドラ —
   は機能しておりテストも通っているため、VitalRouter へ移行しない(YAGNI, cover and modify)

## 背景

- 「このボタンは何をするか」を AI が知れると、探索なしで操作を計画できる
  (Command メタデータの State 公開 = capability discovery)
- 素朴な自然言語説明だけでは実装との乖離に誰も気づけない。
  「実は何も検証していないテスト」を禁じるのと同じ理由(Green means done)で、
  検証に使う情報は嘘をつけない形で公開する必要がある
- 作用の語彙を「プロジェクトに存在するコマンド型の集合」に閉じることで表現の発散も防げる。
  外部コマンド(start)と UI 操作(click)が同じ作用に合流することも型名の一致で表現できる
- VitalRouter は D-005 で採用・D-017 で互換検証済みだが、これまで参照ゼロ(未使用)だった。
  R3(状態の通知)に対し VitalRouter は命令(「これをせよ」)であり、
  UI 操作 → 作用はまさに命令の形。初の実戦投入として筋が良い

## トレードオフ

- コマンド型(StartGameCommand 等)という間接層が1枚増える。ボタン2個の現状には過剰だが、
  作用の公開(本決定の目的)がこの間接層そのものを情報源にするため受け入れる
- 「押した結果どうなるか」(Phase が Playing になる)までは公開しない。
  それはボタンでなく状態機械(GameFlow)の知識であり、結果の検証は従来どおり
  State/Log の観測で行う。アフォーダンスは計画のヒントであって証明ではない
- 将来ネットワーク対戦(リモートコマンド注入)で外部コマンドとゲーム内コマンドを
  VitalRouter で統一する動機が出る可能性はあるが、それはそのときに別の D-xxx で決める

## 状態

採用・実装済み。純C#の `GameCommandRouter`(StartGameCommand / ExitGameCommand →
GameFlow・ExitRequests)をユニットテストし、Godot 層は `Bind` で配線と公開を一本化。
headless E2E で確認済み: `game/ui` に Publishes / Explain が載り、
StartButton クリック → StartGameCommand → Playing、
ExitButton クリック → ExitGameCommand → exit 0。
