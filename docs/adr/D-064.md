# D-064 なぜ任意メソッド呼び出し(PuerTS 的アプローチ)ではなく State/Command/Log 境界なのか

## 決定

AI による動作確認の接点は、スクリプトブリッジ(PuerTS / リフレクション等)による
**任意の内部メソッド呼び出しにせず**、State / Command / Log の**狭い公開境界**に限定する。
これは想定問答(「内部コードを改造せず、AI から C# メソッドを直接呼べばいいのでは?」)
への公式回答を兼ねる。

一言でまとめると:
**任意メソッドが呼べることと、動作を検証できることは別物。検証には
「本番経路・決定論・安定した観測面」が要り、それは内部の設計協力なしには手に入らない。**

## 背景

PuerTS(Tencent)のように C# メソッドをスクリプトから直接呼べるブリッジを埋め込めば、
Statee のような専用フレームワークを組み込まなくても AI がゲームを操作・観測できるように
見える。この突っ込みは自然であり、答えを記録しておく価値がある。

なお「ロジックを叩くだけ」なら確かに Statee は不要で、本プロジェクトもそこは
純C#ロジック層 + xUnit で AI が直接・最厚にテストする方針(テストのレイヤー化)。
Statee が担うのはその上の「**組み上がって動いているゲーム**」の確認である。

反論の骨子:

1. **任意メソッド呼び出しは本番と同じ経路を通らない。**
   内部メソッドを直接叩くと、ゲームループ・入力処理・ECS の Tick 順・メインスレッド制約を
   バイパスでき、実プレイでは到達不可能な状態を作れてしまう。「テストが緑=実際に動く」
   という信頼(Green means done)が壊れる。Statee の Command は本番の入力・コマンド経路を
   通り、State は動いているゲームのスナップショットなので、緑が本番挙動の証拠になる。
2. **再現性はどのみち内部改造が必要。**
   AI の自動確認で本質的に難しいのは「呼べるか」ではなく「再現できるか」。
   freeze / step 実行 / 乱数シード注入(D-040 ほか)はゲームループ側の協力なしに実現できず、
   外付けのスクリプトブリッジでは手に入らない。「内部を触らない案」は実は成立しない。
3. **境界の設計そのものが成果物。**
   State / Command / Log という狭い公開面は、任意メソッドが呼べる広い面と違い、
   リファクタしても壊れにくい安定した契約(テスタビリティの seam)。
   「内部コードの改造」ではなく「公開 API の設計」であり、PuerTS を採る場合も
   JS ランタイム組み込みという内部改造は結局発生する。
4. **プロセス境界と汎用性。**
   MCP → CLI → TCP の構成は headless / CI / 将来のネットワーク対戦
   (リモートコマンド注入)まで同じ形で伸びる。ゲーム内にスクリプト VM を埋め込む方式は
   ゲームごと・エンジンごとに再構築が必要で、「ゲーム非依存の MCP サーバは再ビルド不要」
   という性質が得られない。

## トレードオフ

- 公開境界が狭いぶん、境界にない情報・操作は都度 State / Command を設計して足す必要がある
  (任意メソッド呼び出しなら即席で届く)。その手間を「安定した契約を育てるコスト」として受け入れる
- デバッグ用途で「内部を直接突きたい」場面はあり得るが、それを検証経路に混ぜない

## 状態

採用。
