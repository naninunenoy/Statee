# D-052 マッチングは合言葉ゲートのみ(真のマッチメイキングはやらない)

## 決定

リバーシのネット対戦(D-050)に「対戦成立」の最小限の仕組みとして、**合言葉によるゲート**を
追加する。

- クライアントはネット対戦開始前に合言葉(任意の文字列)を入力する
- サーバは起動時に1つの合言葉(`--room=`。既定値 `reversi`)を持ち、
  一致する合言葉で接続してきたクライアントだけを受け入れる
- 実装は D-050(N-3)で既にあった LiteNetLib の `connectionKey`(`AcceptIfKey`)を
  そのまま流用する。新しい抽象は増やさない
- 座席(黒/白)は引き続き接続順で決まる(D-050 と同じ)。合言葉は「入る権利」の
  フィルタであり、座席や対戦順の交渉には関与しない

**やらないこと**:

- ランダムマッチ・ランク付け・ロビー一覧などの本格的なマッチメイキング
- 複数の合言葉(部屋)を1サーバプロセスで同時に捌くこと(1プロセス = 1合言葉 = 1対局)。
  複数対局を同時に捌きたくなったらサーバプロセスを複数立てる(ポートを分ける)か、
  その時点でマルチルームを設計する
- なりすまし防止・認証(合言葉を知っていれば誰でも入れる)

## 背景

- 前段の会話で「合言葉を事前に決めてマッチングするくらいがいい」という要求があった。
  リバーシは知り合い同士の対戦を想定した最小構成であり、EOS 等のマッチメイキング基盤
  (NETWORK_PLAN.md・D-050 で将来候補として記録のみ)を導入する規模ではない
- 合言葉の入力 UI が必要になったため、Declaree に `LineEdit`(一行テキスト入力)を追加した。
  値を運ぶイベント(`dispatch(eventId, payload)`)までは作らず、ホスト側が `Name`(D-038)で
  Godot コントロールを直接参照して `.Text` を読む非リアクティブな方式にとどめた
  (declaree/README.md の YAGNI 方針どおり、要求されたものだけを足す)

## トレードオフ

- 1プロセス = 1合言葉のため、同時に複数の対戦(複数の合言葉)を捌くには
  サーバプロセスを複数起動する必要がある。マッチメイキングサービスが担う
  「相手を探す」「複数対局を捌く」は完全にスコープ外のまま
- 合言葉は認証ではなく、知っていれば誰でも入れるゲートに過ぎない

## 状態

【決定・実装済み】`game/Reversi.Server`(`--room=`)・`game/Reversi.Godot`
(タイトル画面の合言葉入力 + `connect`/`start` コマンドの `room` 引数)・
`declaree/Declaree`(`LineEdit`)に反映済み。実プロセスで正しい合言葉のみ
ペアリングされ、誤った合言葉は拒否されることを確認した。
