# D-051 マルチインスタンス検証はシナリオ語彙の拡張のみで始める

## 決定

ネット対戦(D-050)の検証のため、Statee.Scenario の Ruby シナリオ語彙を拡張する。
初回スコープは**語彙のみ**とし、以下の3点に限定する。

1. **複数ターゲット接続**: 1つのシナリオから複数の Statee ターゲット(port 違い)へ接続する
   (例: `target :server, port: 5001` / `target :client1, port: 5002`)
2. **宛先指定**: コマンド・state 取得の宛先をターゲット名で指定する(例: `on(:client1) { send ... }`)
3. **クロスインスタンス wait**: 複数ターゲットの State が同時に条件を満たすまで待つ
   (例: server / client1 / client2 の `game/board` が全て一致するまで待つ)。
   これが「同期が取れた」ことの検証の中核であり、コマンドレプリケーション(D-050)の
   受け入れテストになる

**含めないもの**(将来、実需要が生じた時点で設計する):

- 起動オーケストレータ(サーバ + N クライアントの起動・引数注入・終了管理)。
  サーバ1 + クライアント2程度は `tools/` の起動スクリプトで賄う。
  8インスタンス級(NETWORK_PLAN.md の FPS 構想、仮称 Statee.Fleet)が必要になった時に設計する
- MCP(statee-checker)のマルチインスタンス対応
- 複数インスタンスのログを tick 同期で突き合わせるマージビュー

## 背景

- NETWORK_PLAN.md はセッション管理・宛先指定・クロス wait・オーケストレータの4系統を挙げるが、
  リバーシ(サーバ1 + クライアント2・ターン制)に必要な最小はシナリオ語彙だけ
- 「固定待機を書かず State を wait する」(GUIDELINE §3.1)の複数インスタンス版が
  クロスインスタンス wait であり、これは Statee の本質部分。先に固める価値が最も高い

## トレードオフ

- プロセス起動がシナリオの外(スクリプト)にあるため、シナリオ単体では自己完結しない。
  3プロセス程度なら手順の複雑さは許容範囲で、オーケストレータの過剰設計
  (FPS の事情の先取り = D-013 の裏返し)を避ける方を取った

## 状態

【決定・実装済み】(N-6。docs/REVERSI_NETWORK_ROADMAP.md)。
`ScenarioRunner`(src/Statee.Scenario)に `target(name, port:)` / `on(name) { ... }` /
`wait_all([name, ...], path, field, op, value, timeout_ms=nil)` を追加した。
`wait_all` は「列挙した全ターゲットへ同じ wait を順に送る」実装(真の並列ポーリングではない)
だが、確定コマンドの追記ログという前提上、逆行しない条件なら結果は同値になるため
「クロスインスタンスで条件が揃うまで待つ」目的は満たす。起動オーケストレータは作らず、
`tools/run-reversi-network.ps1` を手順メモ代わりに置いた。
