# D-056 RaidBoss を2〜4人可変+合言葉部屋ロビーへ拡張

## 決定

RaidBoss(D-053/D-054)を固定2人から2〜4人の可変人数に拡張し、複数の部屋(合言葉違い)を
1サーバプロセスで同時に扱えるロビー機能を追加する。

- `RaidBoss.Logic.GameLogic` に `GamePhase.Waiting`(参加待ち)を追加し、
  `Start(playerCount)` で2〜4人の参加人数を確定してから `Playing` へ遷移する。
  ボスの反撃対象は人数分を巡回するよう一般化した(乱数を使わない決定論は維持)
- 接続レベルの固定パスフレーズ(D-052の「プロセス起動時に固定の合言葉1つ」)を撤廃し、
  接続後の最初の1通(`create`/`join` コマンド + `room` 引数)で合言葉ごとの部屋へ
  振り分けるハンドシェイクに置き換えた(`Syncee.ManualServerTransport` + 
  `RaidBoss.Server.RoomManager`)
- 部屋の参加者の誰か(通常は作成者)が `start` を送ると、その時点の接続人数で
  `GameLogic.Start`/`TickBundleAuthority.SetExpectedClientCount` を確定する
- RaidBoss.Godot にコード構築(D-033)のロビーUI(参加/部屋を立てるボタン・
  合言葉入力欄・開始ボタン)を追加した

## 背景

- ユーザーからの要望:「ビジュアル・ゲームとしての体裁」「シューティング演出化」
  「最大4人までの参加待機ロビー(部屋作成/参加を合言葉で選べる)」の3点。
  影響度を見積もった結果、ロビー機能はシューティング演出化より前に着手する順とした
  (シューティング化は GameLogic の物理表現を伴う大きな再設計になるため、先に
  可変人数・複数部屋の骨格を安定させる)
- 複数部屋を1サーバプロセスで同時に扱う設計(接続後ハンドシェイクでルーティング)は、
  D-052の「プロセス起動時固定の合言葉1つ」を発展させたもの。プロセスを部屋ごとに
  立てる方式は「アプリ内でjoin/createを選ぶ」体験と噛み合わないため採用しなかった

## トレードオフ

- サーバの `game/raidboss` State は `RoomManager.LastTouchedRoom`(直近に触れられた
  部屋)のみを映す簡易実装。複数部屋を同時に検証したい場合は部屋ごとの State公開が
  必要になるが、1プロセスにつき同時検証は1部屋を想定する割り切りとした
- `TickBundleAuthority` の `expectedClientCount` を後から確定できるよう
  `SetExpectedClientCount` で可変にした。開始前(Waiting中)の誤確定を防ぐため
  デフォルトは到達不能な値(`int.MaxValue`)にしている
- 部屋の「誰が作成者か」は追跡していない。`start` は接続中の誰が送っても同じ結果になる
  冪等な操作として設計し、権限チェックを持たせない(最小規模のサンプルとして許容)
