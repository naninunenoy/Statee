# D-049: 継続入力の注入とフレーム精度リプレイは規約(コアは変えない)

## 決定

1. リアルタイムゲームへの**継続入力(押しっぱなし)の注入**は、ゲームの Godot 層が
   提供するメインスレッドコマンド(慣例名 `tick`)で行う。Statee 本体(`src/`)には
   機能を足さない(D-013, D-043 の原則の延長)
   - 形: `freeze` → `send --command tick --arg frames=N,input=<トークン>+<トークン>`
   - 入力トークンはゲームが定義する(ShootingGame: left/right/up/down/shoot、`-` は無入力)。
     CLI の `--arg` はカンマを複数指定の区切りに使うため、トークンの連結は `+`
2. **入力ログ**は純C#ロジック層が Tick ごとに記録し(凍結後は記録しない)、
   ランレングス圧縮した形を State で公開する(ShootingGame: `game/<name>/inputs` の
   `Runs = ["474 -", "200 shoot", ...]`)。各要素はそのまま `tick` コマンドとして
   再生できる表記にする(RogueGame の ActionLog と同じ「そのまま再生可能」規約)
3. **フレーム精度リプレイ検証**の型: 記録(プレイ)→ 同一シードで再起動 →
   入力ログを先頭から再生 → State(イベント列・座標含む)一致。
   ロジック層は `Replay(seed, inputs)` を提供する(RogueGame の `Replay` と対)

## 背景

- D-048(ShootingGame)の検証の柱「継続入力の注入 API(D-043 の拡張)」を実装した際、
  Statee コアに必要な機能(freeze / step / MainThreadDispatcher / State)はすべて
  揃っており、足りないのは機能ではなく**規約**だった(D-043 と同じ結論)
- 実証: ShootingGame の headless E2E で、記録セッションの入力ログ
  (`474 -,200 shoot,90 up+shoot,150 down+shoot`)を同一シードの新プロセスに再生し、
  全23イベントの発生 Tick・弾座標(float ビット単位)・アイテム位置まで一致した
- 起動からコマンド到達までの「未制御 Tick」は、headless では無入力(`-`)と等価なので、
  入力ログ先頭の `-` ランに対する差分だけ `tick` で埋めれば正確に合流できる

## トレードオフ

- `tick` コマンドは1呼び出しで N Tick を1描画フレーム内で回す(実時間と乖離する)。
  検証・リプレイ用途では利点だが、描画演出を目視したい場合は step / unfreeze を使う
- 入力ログは無限に伸びる(1 Tick = 1 要素。RLE で公開時は圧縮される)。
  5分ゲーム(約 18,000 Tick)では問題にならないため上限は設けない。
  長時間ゲームに流用する場合は要再考
