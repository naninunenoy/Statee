# D-067 Statee の配布は NuGet パッケージを一次形態とする

## 決定

Statee(`src/` の6プロジェクト)を **NuGet パッケージとして配布する**。
リポジトリのクローン・フォークは配布の一次形態にしない。

| パッケージ | 形態 | 備考 |
|---|---|---|
| `Statee.Core` | ライブラリ | ゲームエンジン非依存のコア |
| `Statee.Remote` | ライブラリ | `Statee.Core` にパッケージ依存 |
| `Statee.Generator` | analyzer | `analyzers/dotnet/cs` 配置。`DevelopmentDependency` |
| `Statee.Cli` | dotnet tool | コマンド名 `statee` |
| `Statee.Mcp` | dotnet tool | コマンド名 `statee-mcp` |
| `Statee.Scenario` | dotnet tool | コマンド名 `statee-scenario` |

共通メタデータ(Authors / Version / License / RepositoryUrl / README 同梱)は
`src/Directory.Build.props` に集約する。`src/` 配下のみに効き、tests / game には影響しない。
ライセンスは MIT(リポジトリ直下の `LICENSE`)。

NuGet では運べないもの — `docs/USING.md` 等の掟、`.claude/` のスキル
(`/new-game`・`/verify`)とエージェント定義、雛形 — は
**テンプレートリポジトリ**(GitHub の "Use this template")として別途配る想定。
その具体化はこの決定の範囲外(未着手)。

## 背景

Statee を使ってゲームを作る人にリポジトリごとクローンさせる形は、

- ユーザーのゲームとフレームワークの履歴が混ざる
- フレームワーク更新への追従が git 操作(rebase / merge)になり重い
- サンプルゲームやフレームワークのテストなど、利用者に不要なものが大量に付いてくる

という問題がある。Godot 4.x の .NET 版プロジェクトは通常の csproj なので
`dotnet add package` がそのまま使えるし、Source Generator も CLI(dotnet tool)も
NuGet が標準の配布経路。バージョン番号での追従・固定ができる点もクローンより強い。

## トレードオフ

- 利用者への要求が「.NET 10 SDK」になる(net10.0 統一のため。D-016)
- パッケージのバージョン管理・公開の運用が増える(`VersionPrefix` の更新、`dotnet nuget push`)
- フレームワークをいじりながらゲームを作る開発スタイル(このリポジトリ自体のスタイル)は
  NuGet 参照では回らない。それは monorepo 側(ここ)の役割として残る
- AI 向けの掟・スキルはパッケージに乗らないため、テンプレートリポジトリの整備が
  完了するまで配布物として不完全

## 状態

採用。パッケージング設定済み・`dotnet pack` で6パッケージの生成と内容
(analyzer 配置・パッケージ依存変換・tool インストール動作)を確認済み。
nuget.org への公開、およびテンプレートリポジトリの作成は未着手。
