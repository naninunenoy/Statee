# D-054 RaidBoss の同期方式は決定論ロックステップ(サーバは入力バンドルの確定係)

## 決定

RaidBoss(D-053)のネット同期は、Syncee のサーバ権威モデル(D-050)を維持したまま
以下のロックステップ方式で行う。

1. **固定 Tick 制**でシミュレーションを進める。各クライアントは「実行対象 Tick」を
   `現在 Tick + N`(入力遅延バッファ。数 Tick 先)にして自分の入力コマンド
   (移動方向・攻撃等)を送信する
2. **サーバは物理・当たり判定を持たず、Tick ごとの入力確定係に徹する**。
   両クライアントの「Tick X の入力」が揃うまで待ち、揃った時点で
   「Tick X の確定入力バンドル(両プレイヤー分)」を全クライアントへブロードキャストする。
   これは既存の `AuthorityLog` をそのまま使える(1コマンド=1手 → 1コマンド=1Tick分の
   入力バンドルへ一般化するだけ)
3. **シミュレーション本体はクライアント側で決定論的に実行**する。サーバ・両クライアントは
   同じ純C#ロジック(`RaidBoss.Logic`。D-011 のシード同期と同型)に同じ確定入力バンドルを
   同じ順で流し込み、同じ結果に収束させる。サーバは同じロジックをヘッドレスで並走させ、
   不正入力(範囲外の値等)の検証に使う
4. **途中参加・巻き戻しはスコープ外**。切断検知のみ(D-050 のリバーシと同じ割り切りを踏襲)

## 背景

- 3方式を比較した:
  - **決定論ロックステップ**(採用): 物理演算をサーバ側に複製しなくてよく、
    Syncee の既存コマンド確定モデル(1コマンド=1確定単位)をほぼそのまま拡張できる
  - サーバ権威+状態スナップショット配布: サーバが位置・HP 等の実体を持ち毎フレーム配信する
    MMO/MOBA 方式。途中参加に強いが、D-050 で明示的に見送った「状態スナップショット配布」を
    新規に作り込む必要があり、実装コストが最も大きい
  - リアルタイム随時入力+即時反映(補間・巻き戻し前提): アクションゲームらしいが、
    高頻度送信・補間・巻き戻しという Syncee にとって別の沼に同時に踏み込むことになる
- ロックステップは「両者揃うまで待って確定」という、リバーシにはなかった
  **バッチ確定**パターンを新たに実証できる。じゃんけん案で検討していたパターンを
  本番投入する形になる

## トレードオフ

- 入力遅延バッファ(N Tick 先行入力)により、操作から反映までに固定の遅延が生じる。
  対戦系ほどシビアではない協力プレイなので許容する
- 決定論を壊す非決定要素(浮動小数点演算順序・辞書の列挙順等)混入のリスクは
  D-011/D-049 と同じ注意が必要。物理エンジン(Godot 物理)を使うと決定論保証が難しくなるため、
  当たり判定は当面 `RaidBoss.Logic` 内の純粋計算で行う方針とする
- サーバがヘッドレスで同じシミュレーションを並走させる分、サーバの実装コストは
  「ただの確定係」よりわずかに増える。ただし物理複製がない分、状態スナップショット方式より軽い

## 状態

【決定】設計のみ。実装はローカル(ネット非依存)の決定論ボス戦ループを先に完成させてから、
Syncee 配線(Tick 入力バンドルの確定・配布)を後段のフェーズで足す。
