# D-061 Declaree に差分適用(Reconcile)を導入、Slider はドラッグ中もライブ通知

## 決定

Declaree の更新方式を「全破棄・全再構築(D-035)」から「差分適用(reconcile)」へ変更する。

- **UiDiff.CanPatch(純C#)**: 「描画済み Control を破棄せずに更新できるか」のノード単位判定。
  型・Name・イベント ID(Render 時にクロージャへ固定されるもの)がすべて一致するときだけ true
- **UiRenderer.Reconcile(Godot)**: CanPatch かつ子の個数が一致するノードはプロパティだけ
  その場で更新し、子へ再帰する。不一致のサブツリーだけを作り直して親の同じ位置へ差し込む
  (アンカーは旧 Control から引き継ぐ)
- **Slider は ValueChanged で毎回 dispatch**(D-060 の「確定時のみ通知」を廃止)。
  ドラッグ中もホストが状態を更新 → Reconcile がスライダーを破棄せずパッチするため、
  ドラッグは途切れない
- パッチ側からの書き戻しは `SetValueNoSignal` / `SetPressedNoSignal` を使い、
  dispatch → 更新 → dispatch の通知ループを構造的に断つ
- LineEdit の Text は**宣言が変わったときだけ**上書きする(入力中のユーザーテキストを保護)

## 背景

D-035 は「差分適用は必要が証明されてから導入する」と明記していた。TodoApp で証明が2つ出た:

1. スライダーのドラッグ中に再構築が走ると、スライダー自身が破棄されてドラッグが途切れる。
   D-060 では「確定時のみ通知」で回避したが、ユーザー要求
   「スライダーを動かしている間も文字の大きさに反映」はこの回避策と両立しない
2. 再構築のたびに LineEdit の入力中テキストが消える

## トレードオフ

- イベント ID が変わるノードはパッチできず部分再構築になる(ID にパラメータを埋め込む
  D-060 方式では、対象 Id が変わると行ごと作り直し)。TodoApp の行 UI は Name も
  Id 依存(`TaskRow{id}`)なので、並び替え・追加・削除は自然に再構築パスに落ちる。
  正しさ優先の割り切りで、keyed reconcile(同一 Id の行を移動として扱う)は
  必要が証明されてから
- 子の個数が変わるとコンテナ配下を丸ごと作り直す(挿入位置の推定はしない)。
  ダイアログ開閉(Stack の子 1↔2)はルートごと再構築になるが、頻度が低く問題ない
- Reconcile は Render と対で保守する必要がある(ノード型を足したら ChildrenOf / Patch も
  更新する)。Describe/Render/Reconcile の三点セットが新ノード追加の定型になった
