# D-034: 人間向け検証レポート — expect 語彙・screenshot コマンド・記録デコレータ

## 決定

1. **シナリオ語彙に5つ目 `expect("説明")` を追加する**(HANDOVER「語彙を増やさない」原則の例外)。
   ワイヤには何も送らない。以降の send / wait ステップを、次の expect が現れるまで
   同一の「期待セクション」に束ねる、人間向けメタデータである
2. **スクリーンショットはターゲット側コマンド `screenshot`(args: path)**。
   ゲームが PNG をディスクに保存し、絶対パスを payload で返す。
   base64 画像をワイヤに載せない(行区切り JSON/TOON プロトコル D-018 を汚さない)
3. **記録は `RecordingScenarioClient`(IScenarioClient のデコレータ)**。
   send / wait の各ステップ成功直後に `screenshot` と `state` を追加送信して
   「コマンド・引数・payload・スクショパス・State TOON」を記録する。
   ScenarioRunner の変更は recorder 注入と expect 定義のみ
4. **撮影タイミングは send / wait の成功直後**。wait 完了 = 状態が落ち着いた時点なので
   画が意味を持つ。`state`・`screenshot` 自身のステップは撮影・記録しない
5. **レポートは単一の自己完結 HTML**(CSS インライン、画像のみ `shots/` への相対参照)。
   シナリオ CLI の `--report-dir` 指定時のみ有効。出力先はリポジトリ内に作らない(D-021 の精神)。
   記録・HTML 生成は別 csproj に切り出さず **Statee.Scenario に同居**させる(利用者が
   ScenarioRunner だけで分離理由がない。シナリオ以外からレポートを作る経路が
   現れたら Statee.Report 等への分離を検討する)
6. **レポート実行は --headless なし(窓あり)の Godot で行う**。
   Godot 4 の headless はレンダリングサーバー自体が無くスクショ不可のため。
   既存の headless E2E(/verify)は不変

## 背景

- AI がコマンド駆動で行った動作確認を人間がレビューできる形にしたい。
  「送ったコマンド / 期待する結果 / 実際の結果(スクショ + State TOON)」を並べた
  HTML を出力し、画面と期待の一致を人間が目視判断する
- GUIDELINE §3 のビジュアル検証レイヤーに当たる。幾何述語アサート(`game/ui`)の
  代替ではなく、その上に載る人間レビュー層
- `expect` は「ターゲットに送るなら send で」の原則に当てはまらない
  (ターゲットへ送るものではなく、レポートのための注釈)。よって語彙追加が正しい置き場

## トレードオフ

- 語彙4つ原則の初の例外。歯止めとして「ワイヤに送らないメタデータのみ語彙になり得る、
  ターゲットを動かすものは従来どおり send」と線を引く
- 窓あり実行は CI では回せない。人間レビュー前提の機能なので許容する
- 記録有効時はステップごとに +2 コマンド(screenshot + state)のオーバーヘッド。
  レポート目的の実行に限られるため許容する
- screenshot の path はゲームプロセスの CWD 基準になり得るため、
  クライアント側で絶対パスに解決してから送る
