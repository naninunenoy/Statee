# D-044: 第2サンプルゲームは emoji タイルのターン制ローグライク

## 決定

1. 第2のサンプルゲームとして、ターン制・**完全決定論**(乱数はダンジョン生成の
   シードにのみ使う)のミニローグライク(仮称 RogueGame)を `game/` 配下に追加する
2. 描画は emoji タイル。Noto Color Emoji(OFL)を同梱し、アセットはこのフォント
   1 ファイルのみとする(画像・音・3D モデルは使わない)
3. 境界はスイカゲームと同じ(D-023, D-024, D-031, D-043): ロジックはすべて純C#層、
   Godot 層は emoji 描画と入力 → アクション変換のみ。ゲーム全体を
   「アクション列 → 状態遷移」の決定論的関数として設計する
4. Statee での検証の柱: アクションごとの State 逐次照合 / シード固定の生成再現性 /
   アクション列のリプレイ検証

ゲームの詳細仕様と実装計画は `game/ROGUELIKE_PLAN.md` に置く
(形になった時点で削除される作業ドキュメント)。

## 背景

- スイカゲームはリアルタイム・物理・非決定(D-011)であり、Statee の検証語彙が
  「物理の静止を待つ」型に偏っていた。対極(ターン制・論理・決定論)のターゲットを
  持つことで、状態観測・入力・待機条件の抽象が両極をカバーしているかを試せる
- Entity / アイテム / インベントリ / 手続き生成という、ゲーム開発ヘルパーとして
  再利用性の高い部品(シード管理、グリッド、視界判定、ターンループ)が自然に生まれる
- アセットゼロ制約の中で、emoji タイルは文字描画だけで見た目の情報量を確保できる。
  Godot 4 のテキストサーバーはカラーフォントに対応し、Noto Color Emoji の同梱で
  環境非依存にできる

## トレードオフ

- フォントファイル 1 個だけは「アセット」として同梱する。環境依存(SystemFont)を
  避けるための最小の妥協
- 完全決定論はパズル的な読み合いになる反面、リプレイ性はシードによる生成の変化だけに
  依存する。5 分ゲームなので許容
- サンプルゲームが 2 つになり、フレームワーク変更時の追従コストが増える。
  ただしそれ自体が「Statee がゲーム固有事情に寄っていないか」(D-013)の検査になる
