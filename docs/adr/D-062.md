# D-062 ReorderList のドラッグ状態を宣言(IR)に昇格し、移動中・ドロップ先を可視化

## 決定

ドラッグ並び替えの視覚フィードバック(移動中の行・ドロップ先)を、Godot 層の演出ではなく
**宣言的 UI の状態**として表現する。

- `ReorderList` に `DraggingIndex` / `DropIndex`(null=ドラッグなし)を追加。
  Describe で props(`draggingIndex` / `dropIndex`)に出るため、
  **ドラッグ中の視覚状態そのものを ui/tree の State で検証できる**
- イベントプロトコルを3種類に再編(旧 `"{id}:{from}:{to}"` は廃止。後方互換は持たない):
  - `"{id}:update:{from}:{to}"` — ドラッグ開始・ドロップ先の変化
  - `"{id}:commit:{from}:{to}"` — 確定
  - `"{id}:cancel"` — 不成立(リスト外・同位置で解放)
- ホストは update を受けてドラッグ状態を更新し再宣言する(マウス移動ごとに
  Reconcile(D-061)が走る)。レンダラは宣言に従い、移動中の行を半透明
  (Modulate α=0.4)、ドロップ先の行を緑ハイライトにする
- マウス追従ゴーストは座標が連続量で IR に載せる意味がないため導入しない
  (必要なら Godot 層の演出として将来追加する)

## 背景

- ユーザー要求「ドラッグ中、移動中の要素とドラッグ先の位置を分かりやすく」。
  Godot 層内で完結する演出(`_Draw` でのライン描画等)も可能だったが、
  (1) Declaree を育てるというプロジェクト目的、(2) 見た目と State が同一の宣言から
  導出され「State が正しければ見た目も正しい」を構造的に保証できること、
  (3) D-061 のライブ reconcile がマウス移動ごとの再宣言に耐えると実証済みだったこと、
  から宣言昇格を選んだ
- ドラッグ状態は UI の一時状態なのでロジック層(TodoLogic)には置かず、
  Godot 層のホスト(Main)が持って BuildUi に渡す。ルールではなく表示の関心事

## トレードオフ

- マウス移動でドロップ先が変わるたびに全ツリーの reconcile が走る(実測で問題なし。
  行数が数千になるなら keyed reconcile や部分更新の最適化が必要になる)
- リスト外へ動かしている間のドロップ先は「押下行に留める」仕様にした
  (update イベントの雑音を減らし、宣言が常に有効なインデックスを指す)
- ハイライトの色・透明度はレンダラ固定。テーマ化(宣言側で色指定)は必要が出てから
