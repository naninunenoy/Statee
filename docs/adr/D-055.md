# D-055 サーバ側の接続管理・ブロードキャストは ClientRegistry へ共通化

## 決定

`ReversiAuthority`(D-050)と `RaidBossAuthority`(D-054)がそれぞれ個別に持っていた
以下の配線を、Syncee コアの `ClientRegistry` へ一本化する。

- 接続順の `client-N` 採番
- クライアントからの受信を `clientId` 付きでディスパッチ(`Received` イベント)
- 切断時の登録解除・通知(`ClientDisconnected` イベント)
- 接続中の全クライアントへの送信(`Broadcast`)

確定モデル(`AuthorityLog` の即時確定 / `TickBundleAuthority` のバッチ確定。D-054 で
明示的に統合しないと決めた部分)はこのクラスの外側の責務のまま変えない。
`ClientRegistry` は「接続がある」という前提だけを扱うドメイン非依存の部品。

## 背景

- リバーシ(D-050)・RaidBoss(D-053/D-054)の2つの権威サーバが揃い、`Program.cs`の
  起動骨格とあわせて `ReversiAuthority`/`RaidBossAuthority` の接続管理部分がほぼ
  丸ごと重複していることが分かった(RaidBoss 実装完了直後のふりかえりで発見)
- `libs/Statee.Godot`(D-047)が「Godot 側の配線イディオムを2〜3ゲーム目の重複を見てから
  共通化した」のと同じタイミング感覚で、2例そろった時点での共通化とした
- 確定ロジック自体(AuthorityLog vs TickBundleAuthority)は意味が異なるため統合しない
  (D-054 の判断を維持)。今回共通化したのは、そのどちらを使っていても同じ形をしていた
  「接続そのものの管理」という薄い層だけ

## トレードオフ

- `ClientRegistry` を挟むことで、座席割当(Reversi)のような「接続時に固有の初期化をしたい」
  ケースは `ClientConnected` イベントの購読に間接化される。直接 `IServerTransport` を
  触るより1段抽象が増えるが、テスト(`ClientRegistryTest`)で挙動を担保しているため
  実害は小さいと判断した
- `Program.cs` の起動骨格(--port 系引数のパース・`StateeHost`/`TimeControl`のセットアップ・
  メインループ)は今回は共通化を見送った。State 登録やコマンドの中身がゲームごとに
  大きく異なり、無理に共通化すると呼び出し側にとって不自然なテンプレートメソッドに
  なりかねないため。ADR は起票せず、ARCHITECTURE.md の今後の候補にのみ残す
