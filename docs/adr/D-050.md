# D-050 ネットワーク対戦基盤の方針(サーバ権威 + コマンドレプリケーション + 同期層別出し)

## 決定

リバーシ(docs/REVERSI_ROADMAP.md)を最初の題材に、ネットワーク対戦基盤を以下の方針で作る。

1. **サーバ権威**。権威サーバは **純C#コンソールプロセス**(Godot 非依存)とし、
   ゲームロジック(Reversi.Logic)をそのまま載せる。サーバにも Statee を組み込み、
   権威 State を直接観測できるようにする(Statee ターゲット = Godot という前提はここで壊す)
2. **同期方式はコマンドレプリケーション**。クライアントは着手コマンドをサーバへ送信し、
   サーバが合法性を検証した**確定コマンドの順序付きログ**を全クライアントへ配布する。
   各クライアントは同じ純C#ロジックで確定手を適用する(決定論 D-011・リプレイ規約 D-049 と同型)。
   状態スナップショット配布は採らない(途中参加・復帰用のスナップショットは将来拡張)
3. **同期層はフレームワーク外に別出し**する。Declaree と同じ流儀で新トップレベル
   ディレクトリ(仮称 `syncee/`)に、コア(純C#・Statee 非依存)+ `.Statee` 接着層を置く。
   Statee(`src/`)と同期層は**相互に無依存**
4. **トランスポートは `ITransport` として抽象化**し、実装は
   **フェイク(インプロセス)と LiteNetLib** の2つ。テストの主戦場はフェイクトランスポート
   (freeze / step / 切断注入が全部効く)で、実ソケットのマルチプロセス E2E は薄く保つ。
   ワイヤシリアライズは MemoryPack(TOON は人間/AI 向け観測用、MemoryPack はワイヤ用)
5. **ホスティング形態は当面「自サーバー(専用サーバプロセス)」に固定**する。
   Firebase(確定手ログを追記ログとして載せる)/ EOS(P2Pリレー・ロビー)への差し替えは、
   形態のインターフェースを先に切るのではなく、**「確定手の順序付きログ」という
   プロトコル設計を形態非依存に保つこと**で将来の移植性を担保する(将来の別実装として記録のみ)
6. **切断は初回スコープでは検知まで**。切断 = 対局終了とし、切断イベントを State/Log に公開する。
   フェイクトランスポートでの切断注入テストを含む。再接続(確定手ログの途中からの再送)は
   第2スライス

進め方: リバーシのローカル2人対戦(R-0〜R-5)完成後に着手する。CPU 対戦は作らない。

## 背景

- ARCHITECTURE.md / D-003 の「コマンド駆動はネット対戦(リモートからのコマンド注入)を意識」
  の実証。サーバへのコマンド注入 = Statee のコマンド注入と同型にできるのが最大の資産
- リバーシはターン制・完全決定論のため、補間・予測・許容付き assert(NETWORK_PLAN.md の難題)を
  全て後回しにして、基盤の骨格(権威 + レプリケーション + クロスインスタンス検証)だけを
  純粋に検証できる
- 純C#コンソールサーバは「ロジックは純C#・厚くテスト」(HANDOVER 原則)の延長線上にあり、
  フェイクトランスポート時代のテストコードとサーバ実体が同じものになる

## トレードオフ

- ホスト型(クライアントが権威兼任)よりプロセスが1つ増えるが、権威 State と表示 State の
  検証が濁らない方を取った
- LiteNetLib はターン制には過剰(信頼チャネルしか使わない)だが、将来のリアルタイムゲームへの
  展開意思があるため採用。抽象が UDP 都合に歪まないよう `ITransport` は最小に保つ
- セッション形態(自サーバー / BaaS / P2P)の抽象化は見送った。実装が1つの段階で
  インターフェースを切ると抽象が歪むため、プロトコルの形態非依存性で代替する
- 状態スナップショット配布を持たないため、途中参加・再接続は初回はできない(意図的な縮小)
