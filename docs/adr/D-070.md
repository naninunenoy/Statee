# D-070 ドット絵生成ツール dotee を tools/dotee に置く

## 決定

ドット絵をテキスト定義(`.sprite.txt`: パレット + 文字グリッド)から PNG に変換する
コンソールアプリ **dotee** を `tools/dotee/` に新設する。スコープは
「テキスト → PNG 変換器」に留める(アニメーション・パレット交換等は必要になるまで作らない)。

- 形式: `palette:` 節に「文字 = #RRGGBB(AA)」、`grid:` 節に 1 行 = 1 走査線
- PNG は依存なしの自前エンコーダ(8bit RGBA・zlib 非圧縮ストアド)。
  ドット絵サイズ前提で圧縮率より単純さを優先
- スプライトデータ本体は各ゲーム側(例: `games/MessBreak/art/*.sprite.txt`)に置き、
  ツールはゲーム非依存を保つ
- 確認用の拡大プレビュー(`*_x16.png` 等)は生成物なのでコミットしない(.gitignore)

## 背景

- AI エージェントは画像を直接描けないが、ドット絵は「色の 2 次元配列」なので
  テキストで表現できる。プレースホルダ用スプライトの内製手段が欲しかった
- 最初は PowerShell + System.Drawing で試作したが、PS 5.1 の罠(ハッシュキーの
  大小文字非区別・カンマの結合優先順位・エンコーディング)を多発し、
  System.Drawing は Windows 専用で macOS サポート(build-all.sh)と矛盾する
- テキスト定義はドットの diff レビューがしやすく、パーサを xUnit で
  テストできる(リポジトリの流儀に合う)

## トレードオフ

- 自前 PNG エンコーダは非圧縮なのでファイルが大きい(ドット絵サイズでは無視できる)
- slnx が 1 本増え、`tools/build-all.ps1` の対象になる(-Depth 2 の範囲内)
- ゲーム制作が本格化して本職のドット絵に置き換える際は、.sprite.txt は
  ラフ・仕様書として残す運用を想定
