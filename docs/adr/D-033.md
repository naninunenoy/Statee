# D-033: UI はコード(C#)で構築する。移行するなら「見た目だけ .tscn」

## 決定

1. **UI の構築は当面 C# コードで行う**(現状の `BuildUi()` を維持)
2. 将来移行する場合も全面移行はしない。**「見た目は .tscn、配線は C#」の分割**とする
   - .tscn に置く: ノードツリー、アンカー、テーマ、フォント(視覚的な試行錯誤が効く領域)
   - C# に残す: `Bind`(作用の配線=公開。D-032)、フェーズ購読による表示切り替え(振る舞い)
3. 移行の引き金は次のいずれか。該当したら本決定を更新する
   - レイアウト調整のために「ビルド → headless → `game/ui` 確認」のループが遅いと感じたとき
   - ビジュアル検証(スクリーンショット + 画像分析)を本格化するとき
   - 人間(デザイナー)が UI 制作に参加するとき

## 背景

- Godot の本来のイディオムは「レイアウトはエディタで .tscn、スクリプトは GetNode で参照」であり、
  移行自体は可能。D-032 の `Bind` はノード参照を受け取る形なので配線・公開の仕組みは無傷で使える
- しかしこのリポジトリの主な書き手は AI エージェントで、.tscn はエディタなしの手書き編集が
  エラーを生みやすく、壊れてもコンパイルエラーにならず実行時に沈黙する(GetNode が null)
- コード構築は参照がすべて C# 変数なので、構造を壊せばビルドが赤くなる。
  Green means done と相性が良い
- 現状の UI はボタン2個+ラベル2個で、コード構築のコストは顕在化していない。
  .tscn 化のコスト(文字列参照化、AI が編集しにくい形式)は即座に発生する(YAGNI)

## トレードオフ

- コード構築特有の落とし穴を踏む(実例: アンカーはツリー追加後に設定しないと
  サイズ 0 で確定し中央寄せが効かない。D-031 実装時の実バグ)。
  エディタ経由ならこのクラスの問題は起きない
- UI 要素が増えると `BuildUi` が肥大する。肥大を感じたら分割条件(上記3)を待たず
  部分的な .tscn 化を検討してよい
- `game/ui` State と `click` コマンドは構築方法を区別しないため、
  どちらへ移行してもテスト資産は無傷

## 状態

採用。移行の引き金を踏んだら本決定を更新する。
