# D-065 本番ビルド(ExportRelease)から Statee の外部入口をビルドレベルで除外する

## 決定

Statee の TCP 待ち受け(`StateeTcpServer` / `Statee.Remote`)は開発・検証用の入口であり、
**本番ビルド(Godot の `ExportRelease` 構成)には含めない**。実行時フラグではなく
**ビルドレベル**で除外する:

1. `StateeTcpServer` の起動・停止をエントリポイントの partial ファイル
   `Main.StateeServer.cs` に隔離する
2. `Main.cs` 側は実装なしの partial method で呼ぶ
   (`partial void StartStateeServer(StateeHost host);` / `partial void StopStateeServer();`。
   実装が無い構成では呼び出し自体がコンパイラに消される)
3. csproj で `ExportRelease` のとき `Main.StateeServer.cs` を `Compile Remove` し、
   `Statee.Remote` への `ProjectReference` も条件付きにする

```xml
<ItemGroup Condition="'$(Configuration)' != 'ExportRelease'">
  <ProjectReference Include="..\..\src\Statee.Remote\Statee.Remote.csproj" />
</ItemGroup>
<ItemGroup Condition="'$(Configuration)' == 'ExportRelease'">
  <Compile Remove="Main.StateeServer.cs" />
</ItemGroup>
```

`StateeHost` / State / コマンド登録(Statee.Core)はゲーム内部の配線なので残す。
消えるのは「外部から届く入口」だけ。

## 背景

Statee は AI Agent の動作確認用に localhost の TCP でコマンドを受け付ける。
配布する本番ビルドにこの入口が残ると、プレイヤー環境で外部プロセスから
ゲーム状態の取得・操作(チート・改ざん)ができてしまう。

実行時フラグ(起動引数や `OS.HasFeature`)での無効化も可能だが、
コードとアセンブリが残る以上「フラグの消し忘れ」「引数での再有効化」の余地が残る。
ビルドから落とせば `Statee.Remote.dll` 自体が出力に含まれず、確認も
「本番出力に DLL が無いこと」という機械的なチェックで済む。

Godot .NET のビルド構成は `Debug`(エディタ)/ `ExportDebug` / `ExportRelease` の3つで、
「本番=ExportRelease」という区別が最初から存在するため、これを境界に使う。

## トレードオフ

- 本番相当のバイナリに対して Statee 経由の検証はできない(入口が無いので当然)。
  検証は Debug / ExportDebug で行い、ExportRelease は「入口が無いこと」を確認する
- partial method の間接参照が1枚増える。代わりに `Main.cs` から `#if` を排除できる
- `ExportDebug` には入口が残る。デバッグ配布物にも入れたくない場合は
  csproj の条件を `'$(Configuration)' == 'Debug'` に絞る

## 状態

採用。全サンプルゲーム(SuikaGame / Reversi / ShootingGame / TodoApp / RaidBoss / RogueGame)
に適用済み。ExportRelease の出力に `Statee.Remote.dll` が含まれないことを確認済み。
専用サーバ(`*.Server`)は配布物ではないため対象外。
