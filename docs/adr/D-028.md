# D-028 条件待機は wait コマンド(State フィールドの条件成立まで進める)

- **決定**: `wait` を TimeControl の標準コマンドに追加(GUIDELINE §7-1、D-027 の気づき)。
  - 引数: `path` / `field` / `op`(eq, ne, gt, ge, lt, le)/ `value` / `timeoutMs`
  - **ポーズ中は 1 フレームずつ step**(決定論)、実行中はフレーム進行通知
    (`TimeControl.WaitForNextFrame`)のたびに再評価。固定スリープのポーリングはしない
  - 応答は条件成立後: `Satisfied` / `Frames`(消費フレーム数)/ `Value`(成立時の値)。
    タイムアウトは「条件・現在値」入りのエラー応答(AI が原因を診断できる)
  - 条件評価は `StateeHost.CaptureState(path)`(公開化)+ フィールド名のリフレクション。
    比較は双方が数値に解釈できれば数値、そうでなければ文字列(順序比較は数値のみ)
- **ゲーム側の規約**: `OnFrame()` は **State 更新の後**に呼ぶ。先に呼ぶと wait が
  1フレーム古い State を読む(SuikaGame の _PhysicsProcess で対応済み)
- **トレードオフ**:
  - (+) D-027 の「step 固定フレーム + state ポーリング」が wait 1回に置き換わった
  - (−) 条件はトップレベルのスカラーフィールドのみ(Fruits[i].Y のような入れ子は不可。
    必要になったらパス式を拡張する)
- **E2E で得た知見**:
  - CLI の複数引数はカンマ区切り: `--arg path=game/board,field=Score,op=ge,value=1`
    (ConsoleAppFramework の配列パース。`--arg` の繰り返しは最後だけが残る)
  - ポーズ中に同じ x へ多重 drop すると同座標に重なって出現し、解除直後に即接触・合体する。
    ゲーム仕様としての投下クールダウンは未実装(フレームワークの問題ではない)
- **状態**: ✅ 実装完了(4段階コミット、全63テスト緑)。E2E: `wait Score ge 1` が
  2フレームで成立(空中合体)、タイムアウト時もゲームは再ポーズ状態に復帰
