# D-041: 全 UI 要素へのツリー位置由来の安定 id(UiNodeId)

## 決定

1. Declaree の `UiTree.Describe` は**全記述子**に `id` を付与する。値はツリー位置から
   導出する(ルート `"0"`、子は `"0.1.2"` のようなドット区切りの index パス)
2. id は専用の ValueObject `UiNodeId`(UnitGenerator、string ベース)で表現する。
   導出 API は `UiNodeId.Root` と `Child(int index)`
3. 記述子ツリーの検索 `UiTree.FindById(descriptor, UiNodeId)` を追加する
4. **GUID(v7 含む)は不採用**。理由:
   - Declaree は全再構築方式のため、インスタンスに GUID を振ると再構築のたびに
     ID が変わり、フレームを跨いだ追跡に使えない
   - GUIDv7 はタイムスタンプ埋め込みで実行ごとに値が変わり、決定論(HANDOVER 原則3)を壊す。
     ツリー位置由来なら同じツリー形状 → 同じ id が再実行でも成り立つ

## 背景

- D-038 の `Name` は意味ベースの**任意指定**で、無名の要素は一覧上区別できなかった
  (YAGNI で保留)。大量エンティティ向け State スキーマの検討(docs/NETWORK_PLAN.md)で
  「全オブジェクトを ID で一覧・区別する」必要が見えたため、UI 側の保留を解消した
- ゲームエンティティ側は既に決定論的な連番 ID(FruitId、D-006)があり方針は一貫している

## トレードオフ

- ツリーの**形状が変わると id も変わる**(要素の挿入で後続の index がずれる)。
  フレームを跨いだ同一性追跡には name(意味が安定)を優先し、id は
  「その時点のツリー内で全要素を一意に区別する」用途と割り切る
- `UiSnapshot` は `Describe` を再利用しているため、`ui/tree` State には自動で id が現れる。
  ワイヤ・記述子の形式(Props の string 辞書)は変えていない

## 状態

採用。Declaree ユニットテスト 24 件緑。id 指定クリック等の操作系は必要になったら足す(YAGNI)。
