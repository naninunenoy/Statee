# 開発ガイドライン

[nowsprinting/unity-coding-skills](https://github.com/nowsprinting/unity-coding-skills) のエッセンスを、
本プロジェクト(Godot 4.7 / .NET 10 / xUnit + Shouldly / Statee)向けに翻案したもの。
Unity 固有の記述(Play Mode / asmdef / MonoBehaviour / uGUI 等)は捨て、思想と規律を抽出している。

## 0. 基本理念 — Green means done

コーディングエージェントが自律的に開発を進めるには、
「テストが緑なら完了」というシグナルが信頼できなければならない。
以降のすべての規律は、このシグナルの信頼性を保つためにある。

- アサーションのないテスト・冗長なテスト・不要なテストダブルはシグナルを濁すので作らない
- 固定フレーム数・固定秒数の待機はフレーキーの温床なので禁止(→ 3.1)
- テストは仕様に結合させ、実装に結合させない(リファクタリングで壊れるテストは価値が低い)

## 1. テストレイヤー

対象をもっとも結合度の低いレイヤーに割り当てる。低いレイヤーほど厚く、高いレイヤーほど薄く。

| レイヤー | 対象 | 実行環境 | 密度 |
|---|---|---|---|
| ユニットテスト | 純C#ロジック(Statee.Core / SuikaGame.Logic) | xUnit。エンジン不要 | 最厚。網羅的に |
| 統合テスト | シーン・ノード連携、UI 操作を含む一連の動作 | headless Godot + **Statee**(Command 駆動 → State/Log 検証) | 薄く。オブジェクト間の相互作用に絞る |
| ビジュアル検証 | 実際の描画 | スクリーンショット + AI の画像分析 | 幾何述語で表せないもののみ |
| 手動テスト | 人間の感性(ゲームフィール、音のバランス等) | 人間 | 自動検証可能な項目は置かない |

- 統合テストで機能的アサーションが書けるなら、ビジュアル検証・手動テストに落とさない
- ビジュアル検証で確認できるものを手動テストに落とさない
- **統合テストのハーネスは Statee 自身**。headless Godot を起動し、CLI/Command で駆動して
  State/Log を検証する。フレームワークのドッグフーディングを兼ねる
- 要求カバレッジは「**同レイヤーの証人**」で満たす:
  UI 表示・操作に関する要求はユニットテストでは満たせず、統合テスト以上のレイヤーで検証する

## 2. テスト設計の規律

### 2.1 仕様ベース > 構造ベース

実装構造に結合したテストはリファクタリングで壊れ、価値が急落する。
実装に自信がないとき一時的に構造テストを書くのは可。ただし仕様テストが同じ振る舞いを
カバーしたら削除する。

### 2.2 同値分割 — 1テストメソッド = 1パーティション = 1つの確定した期待結果

- 入力領域を「仕様が同じ結果を定める」クラスに分割し、**パーティションごとに1メソッド**
- **期待値をパラメータ化しない**。入力によって期待結果が変わるなら、それは別パーティション
  → 別メソッドに分ける
- 名前に `Matches…` / `OnlyWhen…` / `DependingOn…`、検証文に「〜のときだけ」が現れたら
  パーティションを混ぜたサイン。分割する
- 同一パーティション内の複数代表値・境界値は 1 つのパラメータ化テスト(`[Theory]`)に畳み込む
- パラメータ化テストの本体に `if` / `switch` を書かない

### 2.3 境界値分析

- 境界値は「そのパーティションの」パラメータ化テストに含める(境界ごとに1メソッド作らない)
- 仕様が境界付近で挙動を変えないなら、パーティションの代表値1つで十分(境界テスト省略可)

### 2.4 その他の技法

- **状態遷移テスト**: FSM を持つ対象は全直接遷移(0-switch)をカバー
- **デシジョンテーブル**: 複数条件の組合せで結果が変わる場合
- **エラー推測**: ゲーム特有の failure パターンから導出
  (連打・同時押し・シーン遷移中の入力・すり抜け・乱数の偏り・数値オーバーフロー・通信断)

### 2.5 命名と検証記述

- メソッドが対象: `<メソッド名>_<条件>_<期待結果>`(メソッド名は翻訳しない)
- メソッドでないもの(シーン・統合・ビジュアル)が対象: `<条件>_<期待結果>`
- 連番 ID を名前に入れない
- 検証は「**観測可能な振る舞いのみ**」を書く。テストフレームワークの属性・同期/非同期の選択・
  フィクスチャ構築方法・テストの存在理由(rationale)は書かない
- 検証に**条件を再記述しない**: メソッド名の `<条件>` セグメントが既に条件を表している。
  検証には期待結果のみを書く(例: `_WhenBossNext_` のテストの検証に
  「IsBossNext が true のとき〜」と書かない)
- 1テスト1検証。例外: 1つの遷移の結果として複数プロパティが同時に正しいべき場合は
  まとめて検証してよい

### 2.6 テストダブル

- **スタブ**(缶詰の応答)/**フェイク**(簡易実装)は Arrange の関心事 — 設計には書かない
- **スパイ**(相互作用の記録)を使う場合のみ設計に明記する
- 依存のエラー系テストの要否:
  - 外部ライブラリ・フレームワーク由来 → スタブで注入してテストする
  - 自プロジェクト内のコード由来 → 信頼してスキップ

### 2.7 乱数(PRNG 依存の対象)

仕様が何を確定しているかで戦略を選ぶ。シード注入(docs/adr/D-003.md)と併用する。

- **PRNG をスタブ化** — 乱数出力→挙動の対応が仕様で確定しているとき。正確な結果を検証
- **範囲検証** — 出力の範囲だけが仕様のとき。繰り返し実行と組み合わせる
- **統計的検証** — 分布が仕様のとき(ダメージ分散・ドロップ率)。サンプリングして統計量を検証
- **特性検証** — 手続き生成物(迷路など)。厳密一致でなく構造的不変条件を検証
  (例:「入口から出口に到達可能」)。**Godot 物理に依存するスイカゲームの検証もこの方針**
  (座標の厳密一致ではなく「合体した・スコアが増えた・ゲームオーバーになった」を検証)

### 2.8 バグ修正とリファクタリング

- バグ修正は**再現テストから**: 修正前に失敗し、修正後に成功するテストを最初に書く。
  再現を確認したらまず再現テストだけをコミットする(テストファーストの整合性を diff で検証可能に)
- 修正が乱しうる隣接挙動に**回帰テスト**を足す(これは修正前から通る)
- リファクタリングは **cover and modify**: 変更前に回帰カバレッジを整えてから実装を変える

## 3. テスト実装の規律

### 3.1 待機は本番側の状態シグナルで

複数フレームにまたがる動作の完了を待つときは、固定フレーム数・固定秒数ではなく、
本番コード側の状態(FSM の状態、`IsPlaying` のような busy シグナル)を待つ。

- Statee の統合テストでは「**State が条件を満たすまで進める**」を基本形にする
- 純C#層では R3 の FakeFrameProvider / FakeTimeProvider で時間を完全に制御する

### 3.2 操作は実際の入力経路で

統合テスト(受け入れテスト)でユーザー操作を再現するときは、ロジックのメソッドを直接呼ばず、
実際の入力パイプラインと同じ経路(Godot の InputEvent 注入)を通す。

- 実ユーザーと同じコードパスを検証できる
- 無効化・非表示・モーダルに隠れた UI への操作が正しく「失敗」する(それがバグ検出になる)
- モーダル/オーバーレイの遮断は**両面で**検証する:
  表示中は背後の要素への操作が失敗すること、**閉じたら操作が復帰すること**の両方

### 3.3 UI レイアウトは幾何述語で検証

「画面内に収まっているか」「要素同士が重なっていないか」「テキストがあふれていないか」は
幾何述語 = 決定的なアサーションで書く(Rect 比較)。CI で人間なしに合否が出る。

- 同じ幾何プロパティにビジュアル検証を重複して作らない
- ビジュアル検証は幾何で表せないもの(可読性・コントラスト・「A が B の右にある」等)に限る
- おおよその位置関係の座標一致検証は brittle なので書かない

### 3.4 オブジェクトの一意識別

同じシーンに同種オブジェクトを複数生成するときは、一意な名前 / 安定した ID を必ず与える。
テストと AI がフレームを跨いで特定インスタンスを追跡できることは State 設計の前提
(docs/adr/D-006.md)。

### 3.5 テストシームは本番ビルドから除外

テストのためのアクセサや注入ポイントを本番コードに足す場合は、条件付きコンパイル
(`#if DEBUG` 等)か `InternalsVisibleTo` で本番ビルドから隔離する。

### 3.6 カテゴリ

xUnit の `[Trait("Category", "...")]` で分類する:
`Integration` / `VisualVerification` / `Acceptance`(要求の証人)/ `IgnoreCI`(タイミング依存等)

## 4. テスタビリティを設計のゲートにする

実装設計の段階でテスト設計を行い、テスタビリティを PASS / WARN / FAIL で評価する。
**FAIL は実装に進まず設計へ差し戻す。**

FAIL 基準:
- ユニットテストの境界が引けない(対象が大きすぎる)
- 状態が隠蔽され外部から検証できない ※ Statee の State 公開はこの問題への構造的な解でもある
- 条件数に対してテストケース数が組合せ爆発する
- 依存が注入できない(static/global 結合、コンストラクタ内での外部依存の `new`)

設計原則:
- 依存はインターフェースで注入し、テストダブルに差し替え可能にする
- **本番コードと同じシームでテストする**: テストから届かせるためだけの
  private → public/internal 昇格はしない。private は private のまま
- 例外は**組合せ爆発の抑制**のみ: 3引数以上はそのヒューリスティック
  (本当のトリガーは独立条件の組合せで公開シーム経由の網羅コストが爆発すること。
  引数が分岐を駆動しないなら適用外)。そのときは 1〜2 入力にだけ依存する凝集した
  サブロジックを独立したクラス/純関数として抽出する。`internal` は抽出の**結果**として
  到達性が不要な場合に選ぶのであって、目的ではない
- リファクタリング時は過剰昇格を巻き戻す: 公開シーム経由で同じ観測結果を検証できる
  internal 直接テストは公開シーム側に移し、不要になった `internal` は `private` に降格する。
  ただし公開経由で観測できない・組合せ爆発の例外に該当するものは残す
- 公開インターフェースは小さく保つ

## 5. コーディング規約

- **後方互換は求められない限り維持しない**。大胆に壊す(互換レイヤーは蓄積して進化を妨げる)
- 1ファイルに公開型は1つ。名前空間はディレクトリ構造と一致させる
- 抽象クラスに `Abstract` / `Base` を付けない。実装側が抽象名をサフィックスに使う
  (`Card` → `AttackCard`, `DefenseCard`)
- enum は単数形 PascalCase、`[Flags]` は複数形
- イベント/Observable の命名: 変化の前は進行形、後は過去分詞(`OpeningDoor` / `DoorOpened`)。
  発火メソッドは `On` プレフィックス、購読側ハンドラは `<発生源>_<イベント名>`
- async: パラメータで受けた `CancellationToken` で `await` する `try` では、
  汎用 `catch` の前に `OperationCanceledException` を rethrow してキャンセルを伝播させる
- インターフェース実装・オーバーライドのドキュメントは `/// <inheritdoc/>`
- **"Why Not" コメント**: 自然な・標準的なアプローチを試して棄却したときは、その理由を
  コメントに残す(人間と AI が同じ袋小路を再探索するのを防ぐ)。
  それ以外の「何をしているか」コメントは書かない
- 守らせたいルールは `.editorconfig` で warning 以上に設定し、**警告ゼロを維持**する
  (エージェントへの規約強制は文書でなく診断で行う)
- 対象バージョン(C# 14 / .NET 10)までのモダンな言語機能・API を使い、古いパターンを使わない

## 6. 開発ワークフロー(テストファースト)

機能実装は以下のステップで進め、各ステップでコミットする。
コミットを分けるのは、テストファーストの整合性を diff で検証可能に保つため。

1. **スケルトン**: 型とシグネチャのみ作成(コンパイル可・空実装。戻り値はリテラルの default)→ コミット
2. **テストファースト**: テストケース設計に基づきテストを実装し、**失敗することを確認** → コミット。
   失敗するはずのテストが通ったら実装に進まず設計を疑う
3. **実装**: プロダクトコードを実装し、**全テスト緑**を確認 → コミット
4. **リファクタリング**: テストの重複排除 → 静的解析警告の解消 → 簡素化。各操作後に全テスト緑を確認 → コミット

既存仕様を**置き換える**スライス(公開 API が変わり既存テストの多数を書き換える場合)では、
スケルトン段階で旧 API を残したまま新 API の骨組みを足して常時コンパイル可能を維持し、
テスト段階で新 API に全面書き換え(旧 API 依存のテストもここで書き換える)、
実装段階で旧 API を撤去する。

テスト重複の定義(ステップ4):
- 真の重複 = **同じ条件 かつ 同じ検証**。どちらか一方だけ同じものは重複ではない
- 同条件のテストを1つのマルチアサートテストに統合しない
- 同一パーティションで引数だけ違うテストはパラメータ化テストに統合する
- **整頓のためにカバレッジを犠牲にしない**: 真の重複か・同一パーティションか疑わしいときは
  両方残す。冗長なテストのコストは保守だけだが、誤った削除・統合は静かにカバレッジを失う

## 7. Statee フレームワークへ還元する要件

このガイドラインの規律から、フレームワーク自体に求められる機能が導かれる:

1. **条件待機コマンド**: 「State が条件を満たすまで進める(タイムアウト付き)」。
   固定フレーム待機の禁止(3.1)を統合テストで実践するための一級機能
2. **State での UI 幾何・操作可否の公開**: Rect・可視性・interactable を State に含め、
   幾何述語アサーション(3.3)を可能にする
3. **入力パイプライン経由の操作コマンド**: ロジック直呼びではなく InputEvent 注入(3.2)
4. **安定 ID**: 全オブジェクトがフレームを跨いで安定した一意 ID を持つ(3.4)
5. **リモート接続機能のビルド分離**: Statee の待ち受け自体がテストシーム(3.5)。
   リリースビルドから除外できる仕組みを持つ
