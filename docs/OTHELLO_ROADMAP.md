# オセロ開発ロードマップ

Statee を使った **2つ目のターゲットゲーム** としてオセロ(リバーシ)を作るまでの計画。
実装計画そのものではなく、フェーズ分割と各フェーズの完了条件を定める。

## 目的(なぜオセロか)

1. **フレームワークの流用性の実証** — スイカゲームで実証済みの Statee を、
   別ゲームに「フレームワーク側の変更なし(または最小限)」で組み込めるかを確かめる。
   PLAN.md の「後の NuGet 化・別ゲームへの流用に備える」の最初の検証になる。
2. **性質の異なるゲームでの検証** — スイカゲームは物理・リアルタイム・連続量。
   オセロはターン制・離散状態・完全決定論。物理も乱数も不要なため、
   headless での State/Command 検証と相性が最も良いクラスのゲームで、
   スイカゲームで作った機能(wait / click / UI State / シナリオ / レポート)の
   ゲーム非依存性を炙り出せる。
3. **将来のネットワーク対戦への布石** — コマンド駆動設計(PLAN.md)の想定する
   「リモートからのコマンド注入」は、2人対戦のオセロが自然な最初の題材になる。
   (ただし本ロードマップのスコープ外。フェーズ O-5 の「その先」として記録のみ)

## 原則(スイカゲームと同じ)

- **Green means done** / テストファーストの4段階(スケルトン → 失敗するテスト → 実装 → リファクタ)で各段階コミット
- 純C#層(Godot 非依存)にロジックを置き、xUnit + Shouldly で厚くテストする
- 固定フレーム・固定秒待機は書かない。State を wait する
- **`src/` にオセロの事情を持ち込まない**(D-013 の対偶)。
  フレームワークに不足を見つけたら、ゲーム非依存の形に抽象化してから `src/` に入れ、ADR を書く

## ソリューション構成(案)

```
game/
├─ Othello.Logic        … 純C#ロジック(盤面・合法手・反転・パス・終局・GameFlow)
├─ Othello.Godot        … Godot プロジェクト(EntryPoint・描画・入力・Statee 組み込み)
└─ (scenarios/)         … Ruby シナリオ(Statee.Scenario で実行)
tests/
└─ Othello.Logic.Tests
```

## 開発フェーズ

| フェーズ | 内容 | 完了条件 |
|---|---|---|
| O-0 | 流用性の棚卸し | SuikaGame.Godot の Statee 組み込みコードを調査し、ゲーム非依存部分(接続待ち受け起動・State 登録・screenshot 等)の共有方法を ADR で決定 |
| O-1 | オセロロジック(純C#) | 盤面・合法手列挙・石の反転・パス・終局判定・スコアが動き、ユニットテストが全て緑 |
| O-2 | Godot 統合(最小) | headless で起動し、ping / state / logs / quit が通る。`game/board`(8x8 盤面)と `game/turn` を State 公開 |
| O-3 | コマンド駆動 | `place <x> <y>`(非合法手はエラー)、`start` で対局開始。headless で1局を最後まで打ち切れる |
| O-4 | UI・ビジュアル | タイトル → 対局 → 結果の画面遷移、`game/ui` State、`click` での着手。窓ありでスクショが撮れる |
| O-5 | AI 自動動作確認 | Ruby シナリオで「定石数手 → パス発生局面 → 終局・勝敗判定」を検証し、`--report-dir` で HTML レポートが出る |

### O-0: 流用性の棚卸し(フレームワーク側の宿題を先に確定する)

オセロを書き始める前に、スイカゲームで書いた「ゲーム側の Statee 組み込みコード」のうち
2つ目のゲームでも丸ごと必要になるものを特定する。

- 対象候補: Main.cs の接続待ち受け起動、freeze/step のメインスレッドディスパッチ(D-025)、
  screenshot コマンド、SceneState / UiState の骨格
- 判断: ① `src/Statee.Godot`(Godot 依存の共有層)として抽出する / ② コピーして育て、
  3つ目のゲームで抽出する(Rule of Three)— のどちらかを ADR(D-0xx)で決定
- **完了条件**: ADR が書かれ、O-2 で何をコピーし何を参照するかが明確になっている

### O-1: オセロロジック(純C#)

Statee もGodot も登場しない。オセロのルールだけを実装する。

- 盤面表現(8x8、黒/白/空)、初期配置
- 合法手列挙・着手による反転・手番交代
- 合法手なし → 自動パス、双方パス → 終局、石数カウント → 勝敗
- スイカゲームの `GameFlow` / `GamePhase` に相当する対局フロー(Title → Playing → Result)
- **テスト観点**: 既知局面での合法手一致、反転の全8方向、パス強制局面、64手完走、
  「角に置ける局面」等の性質テスト。ここが最厚のテスト層になる

### O-2: Godot 統合(最小)

- Othello.Godot プロジェクト作成、headless 起動、Statee 接続待ち受け(O-0 の決定に従う)
- State 公開: `game/board`(8x8 の盤面。TOON で人間可読に)、`game/turn`(手番・手数)、`game/scene`
- 描画はまだ最小(または無し)で良い。**完了条件は headless での疎通**

### O-3: コマンド駆動

- `start` / `place <x> <y>` コマンド。非合法手は明確なエラーメッセージで拒否
- ターン制なので freeze/step は本質的に不要だが、既存機能がそのまま動くことは確認する
- **完了条件**: CLI から1局を最後まで打ち、`game/board` の終局状態と勝敗が観測できる

### O-4: UI・ビジュアル

- 盤・石の描画、タイトル画面(開始/終了)、結果画面(勝敗・石数)
- `game/ui` State(幾何・テキスト・Publishes/Explain、D-031/D-032 と同型)
- `click` で盤のマスを叩いて着手できる(座標 → マス変換はロジック側でテスト)
- **完了条件**: 窓あり実行で人間がプレイでき、screenshot コマンドが盤面を撮れる

### O-5: AI 自動動作確認(ドッグフーディング)

- Ruby シナリオ例: ① 定石数手を打ち盤面を expect で検証、
  ② パスが発生する局面を再現しパスの State を検証、③ 終局まで打ち切り勝敗を検証
- `--report-dir` 付きで実行し、各手のスクショ + 盤面 State が並ぶ HTML レポートが出る
- **完了条件**: AI Agent(statee-checker)が MCP 経由でシナリオを完遂し、レポートを提示できる

### その先(スコープ外・記録のみ)

- 簡易 AI(合法手ランダム / 貪欲)を相手にした自動対局シナリオ
- リモートからのコマンド注入による2人対戦(PLAN.md のネットワーク対戦構想の実証)

## リスク・注意点

- **O-0 を飛ばさない**。コピペで始めると `src/` に入れるべき共有コードの境界が永久に曖昧になる。
  逆に、抽象化を焦って `src/` にオセロの都合を持ち込まない(D-013)
- スイカゲームと違い物理・乱数がないため、シード注入・IsSleeping 等の機能は使わない。
  「使わない機能がある」こと自体は問題ではない(汎用フレームワークとして正常)
- 着工時に PLAN.md へ本ロードマップの参照とフェーズ状況を追記し、O-0 の判断を ADR 化する
