# 引き継ぎ文書 — このプロジェクトの思想と歩き方

このプロジェクトを引き継ぐ人が**最初に読む1枚**。
細かい仕様は他のドキュメントにある。ここには「何を大事にしているか」と
「迷ったときにどう判断するか」だけを書く。

## 読む順番

1. この文書(思想と判断基準)
2. `docs/PLAN.md` — 何を作っているか(アーキテクチャ・フェーズ)
3. `docs/GUIDELINE.md` — どう作るか(テスト設計・規約。**コードを書く前に必読**)
4. `docs/adr/README.md` — なぜそうしたか(決定の索引。必要な D-xxx だけ読めばよい)

## このプロジェクトは何か(30秒版)

**AI にゲームの動作確認をさせるためのフレームワーク**を作っている。
ゲームが State(状態)・Command(操作)・Log(履歴)の3つを外部に公開し、
AI が「コマンドを送る → State/Log を読む → 期待と照合する」を繰り返せば
人間なしで動作確認が完了する。スイカゲームはその実証用サンプルにすぎない。

## 絶対に守る5つの原則

これを崩す変更は「便利そうでも」入れないこと。崩したくなったら、
まず対応する D-xxx を読んで、それでも崩すなら新しい D-xxx を書いてから。

### 1. Green means done(テストの緑が完了の証明)

テストが緑なら完了、赤なら未完了。この信号を信頼できる状態に保つことが最優先。
だから「たまに失敗するテスト」「実は何も検証していないテスト」を絶対に残さない。
機能実装は必ず「スケルトン → 失敗するテスト → 実装 → リファクタ」の4段階で進め、
各段階でコミットする(D-014)。**テストを先に書けない実装は、設計がおかしい合図**。

### 2. ロジックは純C#に置き、Godot 層は薄く保つ

ルール・判定・状態機械は Godot に依存しない純C#プロジェクト
(`src/Statee.Core`、`game/SuikaGame.Logic`)に置き、xUnit で厚くテストする。
Godot 層(`*.Godot`)はエントリポイント・描画・入力・物理の配線だけ。
「このロジック、Godot 層に書いた方が楽だな」と思ったら止まること。
楽なのは今日だけで、明日からテストできなくなる(D-023, D-024)。

### 3. 決定論を壊さない

AI の動作確認は再現できなければ意味がない。だから:
- 時間は外から与える(`Tick(delta)`)。ロジック内で `DateTime.Now` / `Task.Delay` /
  自前タイマーを使わない(D-023)
- 乱数はシード注入(`--seed=`)
- **固定フレーム数・固定秒数の待機を書かない**。「State が条件を満たすまで待つ」
  (`wait` コマンド、D-028)が唯一の待ち方

### 4. フレームワークにゲームの事情を持ち込まない

`src/` は汎用フレームワーク。スイカゲーム固有の概念(フルーツ、合体、スコア)を
`src/` に1つでも入れたら負け(D-013)。フレームワークの検証は
`sandbox/PingTarget.Godot`(最小ダミー)で行う。
逆方向も同じ: MCP サーバーは CLI を起動するだけの汎用品に保つ。
ゲーム依存の実装は CLI 以下に置く(D-001。理由: MCP はセッション中に再ビルドできない)。

### 5. 記録してから変える

設計判断を下した・変えたら `docs/adr/D-0xx.md` に「決定 / 背景 / トレードオフ / 状態」を
書き、索引(`docs/adr/README.md`)に1行足す。**過去の決定を変えるのは自由**だが、
「なぜ元の決定があったか」を読まずに変えるのは禁止。
境界設計で悩んだ過程は `docs/adr/notes/` に書き捨てでよい(確定したら D-xxx へ昇格)。

## 迷ったときの判断基準(チートシート)

| 迷い | 答え |
|---|---|
| この機能、今つくるべき? | 使う実装が無いなら作らない(YAGNI)。必要になったスライスで足す |
| 後方互換を残すべき? | 求められない限り残さない。警告ゼロを維持 |
| 待機どうする? | 固定時間は禁止。`wait` か State 再取得で条件成立を待つ |
| Godot API をどのスレッドから触る? | メインスレッドのみ。コマンドハンドラはソケットスレッドなので `RegisterMainThreadCommand`(D-025) |
| Godot 内の async コード | **await には必ず `ConfigureAwait(false)`**。無いと自己デッドロックする(D-025 の実バグ) |
| ワイヤプロトコル変えたい | 両端(Statee.Remote と Statee.Cli)を同時に変える。片側だけは禁止(D-018) |
| テストが書きにくい | 実装ではなく境界の設計を疑う。「1メソッド1パーティション」(GUIDELINE) |
| 大きい機能をどう進める? | 縦切りスライス(ping スライス D-018 が手本)。全層を最小で貫通させてから太らせる |

## ハマりどころ(先人が全部踏んだ)

- **Godot は .NET 版バイナリ必須**。標準版では C# が動かない。パスは CLAUDE.md 参照
- **初回の `--headless --import` は完了後にクラッシュする**(exit 0xC0000005)。
  インポートは成功しているので exit code を無視してよい(D-016)
- **MCP サーバー実行中は `Statee.Mcp` がビルドできない**(exe ロック)。
  ソリューション全体ではなく対象プロジェクトだけビルドする
- **CLI の複数引数はカンマ区切り**: `--arg k1=v1,k2=v2`。`--arg` を繰り返すと最後しか残らない
- **ProcessMode は子ノードに継承される**。ポーズ対象は明示的に `Pausable` を宣言(D-026)
- **物理コールバック(`body_entered`)中のノード操作**は、削除 `QueueFree` /
  追加 `CallDeferred` で次フレームへ(notes/suika-physics-boundary.md)
- `.cs` のフォーマットは hooks が自動でやる。手動整形も CI 待ちも不要

## 開発の1日の流れ(型)

1. `docs/PLAN.md` の「現在のマイルストーン」を見て、次のスライスを決める
2. スケルトン(型・シグネチャだけ)を書いてコミット
3. 失敗するテストを書いてコミット(このテストが仕様書になる)
4. 実装してテストを緑にしてコミット
5. リファクタしてコミット。**全テスト緑・警告ゼロ**を確認
6. E2E が必要な変更なら `/verify`(headless Godot で実機確認)
7. 設計判断があれば D-xxx を書く。PLAN.md のマイルストーンを更新する

## やらないことリスト

- 物理の決定論化(座標の完全一致検証)はしない。検証は「合体した・スコアが増えた」等の
  State/Log ベースで行う(D-011)
- リポジトリ内にログ・トレースファイルを作らない。トレースは `STATEE_TRACE` で
  リポジトリ外へ(D-021)
- Ruby シナリオの語彙を増やさない。send / state / wait / assert の4つで表現できないなら、
  それはターゲット側コマンド(send で呼ぶ)にすべき機能
