# リバーシ開発ロードマップ

Statee を使ったターゲットゲームとしてリバーシを作るまでの計画。
実装計画そのものではなく、フェーズ分割と各フェーズの完了条件を定める。

ゲームモードは2つ: **ローカル2人対戦(1画面で交互に着手)** と **ネット対戦**。
CPU 対戦は作らない。本ロードマップのスコープはローカル2人対戦の完成(R-0〜R-5)までで、
ネット対戦は docs/adr/D-050(基盤方針)・D-051(検証語彙)に従い「その先」として続ける。

## 目的(なぜリバーシか)

1. **フレームワークの流用性の実証** — スイカゲームで実証済みの Statee を、
   別ゲームに「フレームワーク側の変更なし(または最小限)」で組み込めるかを確かめる。
   ARCHITECTURE.md の「後の NuGet 化・別ゲームへの流用に備える」の最初の検証になる。
2. **性質の異なるゲームでの検証** — スイカゲームは物理・リアルタイム・連続量。
   リバーシはターン制・離散状態・完全決定論。物理も乱数も不要なため、
   headless での State/Command 検証と相性が最も良いクラスのゲームで、
   スイカゲームで作った機能(wait / click / UI State / シナリオ / レポート)の
   ゲーム非依存性を炙り出せる。
3. **ネットワーク対戦基盤の題材** — コマンド駆動設計(ARCHITECTURE.md)の想定する
   「リモートからのコマンド注入」の最初の実証台。ローカル2人対戦の完成後、
   D-050 のコマンドレプリケーションでネット対戦化する。

## 原則(スイカゲームと同じ)

- **Green means done** / テストファーストの4段階(スケルトン → 失敗するテスト → 実装 → リファクタ)で各段階コミット
- 純C#層(Godot 非依存)にロジックを置き、xUnit + Shouldly で厚くテストする
- 固定フレーム・固定秒待機は書かない。State を wait する
- **`src/` にリバーシの事情を持ち込まない**(D-013 の対偶)。
  フレームワークに不足を見つけたら、ゲーム非依存の形に抽象化してから `src/` に入れ、ADR を書く

## ソリューション構成(案)

```
game/
├─ Reversi.Logic        … 純C#ロジック(盤面・合法手・反転・パス・終局・GameFlow)
├─ Reversi.Godot        … Godot プロジェクト(EntryPoint・描画・入力・Statee 組み込み)
└─ (scenarios/)         … Ruby シナリオ(Statee.Scenario で実行)
tests/
└─ Reversi.Logic.Tests
```

ネット対戦化の際に権威サーバ(純C#コンソール)と同期層(`syncee/`)が加わる(D-050)。

## 開発フェーズ

| フェーズ | 内容 | 完了条件 |
|---|---|---|
| R-0 | 流用性の棚卸し | SuikaGame.Godot の Statee 組み込みコードを調査し、ゲーム非依存部分(接続待ち受け起動・State 登録・screenshot 等)の共有方法を ADR で決定 |
| R-1 | リバーシロジック(純C#) | 盤面・合法手列挙・石の反転・パス・終局判定・スコアが動き、ユニットテストが全て緑 |
| R-2 | Godot 統合(最小) | headless で起動し、ping / state / logs / quit が通る。`game/board`(8x8 盤面)と `game/turn` を State 公開 |
| R-3 | コマンド駆動 | `place <x> <y>`(非合法手はエラー)、`start` で対局開始。headless で1局を最後まで打ち切れる |
| R-4 | UI・ビジュアル | タイトル(モード選択)→ 対局 → 結果の画面遷移、`game/ui` State、`click` での着手。窓ありでスクショが撮れる |
| R-5 | AI 自動動作確認 ✅ | Ruby シナリオで「定石数手 → パス発生局面 → 終局・勝敗判定」を検証し、`--report-dir` で HTML レポートが出る |

### R-0: 流用性の棚卸し(フレームワーク側の宿題を先に確定する)

リバーシを書き始める前に、スイカゲームで書いた「ゲーム側の Statee 組み込みコード」のうち
2つ目以降のゲームでも丸ごと必要になるものを特定する。

- 対象候補: Main.cs の接続待ち受け起動(StateeHost + StateeTcpServer + MainThreadDispatcher の配線)、
  screenshot / click / key コマンドの登録(実装はゲーム非依存の Godot コード)、
  SnapshotStateProvider(現状 Main.cs のプライベートクラスだが完全に汎用)、
  SceneState の骨格、`--port=` / `--seed=` の引数パース
- UI State は対象外: 手書き UiState は D-036 で廃止済みで、`ui/tree` の公開は
  すでに共有層(Declaree.Statee の UiStateProvider)にある
- 判断: ① `src/Statee.Godot`(Godot 依存の共有層)として抽出する / ② コピーして育て、
  次のゲームで抽出する(Rule of Three)— のどちらかを ADR(D-0xx)で決定
- **完了条件**: ADR が書かれ、R-2 で何をコピーし何を参照するかが明確になっている

### R-1: リバーシロジック(純C#)

Statee も Godot も登場しない。リバーシのルールだけを実装する。

- 盤面表現(8x8、黒/白/空)、初期配置
- 合法手列挙・着手による反転・手番交代
- 合法手なし → 自動パス、双方パス → 終局、石数カウント → 勝敗
- スイカゲームの `GameFlow` / `GamePhase` に相当する対局フロー
  (Title(モード選択)→ Playing → Result)。モードは「ローカル2人 / ネット対戦」の2値で持ち、
  ロジック層は着手コマンドの供給元を知らない(ローカルでもネットでも同じ `place` が来るだけ)
- **テスト観点**: 既知局面での合法手一致、反転の全8方向、パス強制局面、64手完走、
  「角に置ける局面」等の性質テスト。ここが最厚のテスト層になる

### R-2: Godot 統合(最小)

- Reversi.Godot プロジェクト作成、headless 起動、Statee 接続待ち受け(R-0 の決定に従う)
- State 公開: `game/board`(8x8 の盤面。TOON で人間可読に)、`game/turn`(手番・手数)、`game/scene`
- 描画はまだ最小(または無し)で良い。**完了条件は headless での疎通**

### R-3: コマンド駆動

- `start` / `place <x> <y>` コマンド。非合法手は明確なエラーメッセージで拒否
- ローカル2人対戦は「同じ画面・同じ入力経路で黒番と白番が交互に `place`」であり、
  ロジック上は追加機能ではない(手番管理が拒否すべき手を拒否するだけ)
- ターン制なので freeze/step は本質的に不要だが、既存機能がそのまま動くことは確認する
- **完了条件**: CLI から1局を最後まで打ち、`game/board` の終局状態と勝敗が観測できる

### R-4: UI・ビジュアル

- 盤・石の描画、タイトル画面(**モード選択: ローカル2人 / ネット対戦** と終了。
  ネット対戦は本ロードマップでは選択肢を出すだけで「未実装」表示で良い)、
  結果画面(勝敗・石数)
- UI は Declaree で宣言する(D-035/D-036。手書き UiState は廃止済み)。
  State は `ui/tree`(幾何 Rect・Explain・Name 込み)として自動公開される
- `click` で盤のマスを叩いて着手できる。マスに安定 Name(例: `Cell_3_4`)を振れば
  `click --arg name=...` で座標決め打ち不要(D-038)。盤を Declaree でなく
  Node2D 描画にする場合のみ、座標 → マス変換をロジック側でテストする
- **完了条件**: 窓あり実行で人間が2人交互にプレイでき、screenshot コマンドが盤面を撮れる

### R-5: AI 自動動作確認(ドッグフーディング)

- Ruby シナリオ例: ① 定石数手を打ち盤面を expect で検証、
  ② パスが発生する局面を再現しパスの State を検証、③ 終局まで打ち切り勝敗を検証
- `--report-dir` 付きで実行し、各手のスクショ + 盤面 State が並ぶ HTML レポートが出る
- **完了条件**: AI Agent(statee-checker)が MCP 経由でシナリオを完遂し、レポートを提示できる

**完了**: `game/Reversi.Godot/scenarios/` に3シナリオを実装。
`opening.rb`(定石4手 → 盤面 State 検証)、`pass.rb`(黒がパスに追い込まれる18手 →
MoveLog のパス記録と手番維持を検証)、`fullgame.rb`(60手完走 → 白の勝ち・黒19白45を検証)。
いずれも headless で `dotnet run --project src/Statee.Scenario -- run --script ... --port 9320`
で exit 0 を確認済み。`fullgame.rb` は窓あり起動 + `--report-dir` で
スクショ付き HTML レポートの生成も確認済み。

### その先(本ロードマップのスコープ外)

- **ネット対戦モードの実装** — D-050(サーバ権威 + コマンドレプリケーション + `syncee/`)、
  D-051(マルチインスタンス検証語彙)に従う。NETWORK_PLAN.md も参照
- CPU 対戦は**作らない**(決定済み。対戦相手はもう一人の人間かリモートの対戦相手)

## フレームワーク側のテコ入れ候補(事前評価)

- **Statee(src/)**: ローカル2人対戦までは変更不要の見込み。state / wait / freeze / step /
  logs / screenshot / シナリオ / レポートはすべてゲーム非依存で src/ にある。残る宿題は
  「ゲーム側ボイラープレートの共有」であり、それが R-0 そのもの。
  ネット対戦化の段階で D-051 の検証語彙(宛先指定・クロスインスタンス wait)が要る
- **Declaree**: 盤面を Declaree UI で組む場合のみ語彙が不足する
  - `GridContainer` 相当が無い(8x8 盤の自然な表現)。`VBox` × `HBox` 8本で代用は可能
  - 石の描画に使える `ColorRect` / `TextureRect` が無い。`Button("●")` のテキスト表現で
    最小スタートは可能
  - 追加する場合は「ターゲットが要求したものだけを足す」(declaree/README.md の YAGNI 方針)
    に従い、R-4 着工時に判断する。盤を Node2D 描画にすれば Declaree のテコ入れはゼロ

## リスク・注意点

- **R-0 を飛ばさない**。コピペで始めると `src/` に入れるべき共有コードの境界が永久に曖昧になる。
  逆に、抽象化を焦って `src/` にリバーシの都合を持ち込まない(D-013)
- スイカゲームと違い物理・乱数がないため、シード注入・IsSleeping 等の機能は使わない。
  「使わない機能がある」こと自体は問題ではない(汎用フレームワークとして正常)
- 着工時に ARCHITECTURE.md へ本ロードマップの参照とフェーズ状況を追記し、R-0 の判断を ADR 化する
