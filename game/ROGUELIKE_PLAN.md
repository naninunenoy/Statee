# RogueGame 計画(仮称。形になったらこのファイルは削除する)

第2サンプルゲーム: emoji タイルのターン制ミニローグライク(→ docs/adr/D-044.md)。
スイカゲームと検証特性が対極(ターン制・論理・決定論)のターゲットとして、
Statee / ゲーム開発ヘルパーの共通要素を別方向から蓄積する。

## ゲーム仕様

- **構成**: 5 フロアを降りて最下層の 💎 を取り、階段を戻って地上へ帰還したら勝ち
  (5 分程度のワンセッション)
- **帰路の演出**: 💎 取得で各フロアに敵が増援され、全敵が追跡モードに覚醒する。
  往路で温存したアイテムを帰路で使う駆け引きを作る
- **戦闘**: bump-to-attack。HP + 攻撃力のみ、**乱数なしの完全決定論**
  (乱数はダンジョン生成のシードにのみ使う)
- **アイテム**: 3〜4 種(🧪 回復 / 🗡️ 武器強化 / 🗝️ 鍵と施錠扉 等)+ インベントリ
  (拾う / 使う)
- **敵 AI**: 視界内なら 1 歩近づく直進追跡、視界外は静止。決定論的
- **視界**: 探索済み地形は記憶表示、Entity(敵・アイテム)は視界内のみ表示。
  視界判定は敵 AI の「見えたら追跡」と共用する
- **生成**: 部屋+通路の古典的生成。シードから完全再現可能
- **描画**: emoji タイル。Noto Color Emoji(OFL)をフォントとして同梱し、
  グリッドに文字を置くだけで描画する。画像・音・3D アセットは使わない

## アーキテクチャ

- スイカゲームと同じ境界(D-023, D-024, D-031, D-043): 純C#層に
  生成・ターン処理・戦闘・インベントリ・視界のすべてを置き、
  Godot 層は emoji 描画とキー入力 → アクション変換のみ
- ゲーム全体が「アクション列 → 状態遷移」の決定論的な関数になるよう設計する
- 帰路の増援・覚醒のため、離脱したフロアの状態を保持する(フロアは使い捨てない)

## Statee での検証の柱

1. アクション(移動・攻撃・拾う・使う)ごとの State 逐次照合(D-043 の入力 API の主用途)
2. シード固定でのダンジョン生成再現性(headless で同一シード → 同一 State)
3. プレイログ(アクション列)のリプレイ検証 — 記録したアクション列の再生が
   同一結果になることの確認。決定論設計の強みを検証機能として昇華する
