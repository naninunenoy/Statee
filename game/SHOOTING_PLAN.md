# ShootingGame 計画(仮称。形になったらこのファイルは削除する)

第3サンプルゲーム: emoji の横スクロールシューティング(→ docs/adr/D-048.md)。
**リアルタイム × 完全決定論**という第3の検証特性象限を埋め、これまで遊んでいた
VitalRouter(多対多イベント)・Arch(大量エンティティ)を本来の形で活用する。

| サンプル | 時間 | 決定論 | 主な検証語彙 |
|---|---|---|---|
| SuikaGame | リアルタイム | 非決定(Godot 物理) | 物理の静止待ち・UI |
| RogueGame | ターン制 | 完全決定論 | アクション逐次照合・リプレイ |
| ShootingGame | リアルタイム | **完全決定論**(固定Δt+自前運動) | 継続入力・イベントログ・フレーム精度リプレイ |

## ゲーム仕様

- **構成**: 横スクロール STG。3ウェーブ+ボスの1本道、1セッション約5分。
  スコアとクリア(ボス撃破)/ゲームオーバー(残機0)を持つ
- **決定論**: 論理は固定タイムステップ(60Hz)の `Tick()` で進む。
  弾・敵の運動は数式(等速・サイン波・自機狙い)で **Godot 物理を使わない**。
  乱数はシード由来の1系統のみ(ウェーブ内の出現タイミング・位置に使う)
- **自機**: 4方向移動+ショット(押しっぱなし連射・発射レート制限)。被弾で残機-1、
  短い無敵時間。残機3
- **敵**: 3〜4種(直進 👾 / サイン波 🛸 / 自機狙い弾を撃つ 🦑)+ボス 🐙
  (HP バー、フェーズで弾幕が変わる)
- **アイテム**: 敵撃破でまれに ⭐(ショット強化。被弾で1段階ダウン)
- **当たり判定**: 円 vs 円の自前判定(中心距離)。判定は見た目より小さめ
- **描画**: emoji タイル+同梱 Noto Color Emoji サブセット(RogueGame と同じ方式)。
  自機 🚀 / 敵弾 🔴 / 自弾 🔹 / 爆発 💥。画像・音・3D アセットは使わない

## アーキテクチャ

- 境界はこれまでと同じ: 純C#層(`game/ShootingGame.Logic`)に運動・衝突・出現・
  スコア・残機のすべて。Godot 層は描画と入力→入力状態の変換のみ
- **エンティティは Arch(ECS)**: 弾・敵・アイテムを Entity + Component
  (Position / Velocity / Collider / Team 等)で持ち、システムが Tick ごとに走る。
  大量エンティティで D-004 の採用を初めて本格活用する
- **ゲーム内イベントは VitalRouter**: `EnemySpawned / EnemyDestroyed / PlayerHit /
  PowerUpCollected / WaveCleared / BossPhaseChanged` 等を Publish し、
  スコア係・ウェーブ進行係・State 更新係・演出係がそれぞれ購読する(多対多)。
  SuikaGame の switch 1個(D-005 の反省)とは違う本来の使い方
- **入力は「フレームごとの入力状態」**: `InputState(Left/Right/Up/Down/Shoot の bool)`
  を毎 Tick 論理へ渡す。キー押下イベントではなく状態のスナップショットにすることで、
  記録・注入・リプレイが単純になる

## Statee での検証の柱

1. **継続入力の注入 API**(D-043 の拡張): 「入力状態を設定して N フレーム step する」を
   freeze/step と組み合わせて実現する。エージェントがリアルタイムゲームを
   フレーム精度でプレイできることの実証
2. **イベントログの State 公開**: VitalRouter の **interceptor** で全イベントを横取りして
   リングバッファに記録し、State として公開する。RogueGame の ActionLog(入力の記録)と
   対になる「出来事の記録」。`wait` の条件(例: EnemyDestroyed が N 件)にも使う
3. **フレーム精度リプレイ**: シード+「Tick ごとの InputState 列」の再生が同一結果に
   なることの検証。ターン制(RogueGame)で確立した型をリアルタイムへ拡張する
4. シード固定のウェーブ生成再現性(headless で同一シード → 同一イベント列)
