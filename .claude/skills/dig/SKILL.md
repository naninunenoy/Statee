# dig — 設計・意思決定の深掘り質問

プラン・設計・意思決定のあらゆる側面について、共通の理解に達するまでユーザーに徹底的に質問する。
甘い回答や曖昧な返答には突っ込む。設計ツリーの各分岐をたどり、判断の依存関係を一つずつ解決する。
「幅」より「深さ」を優先する。新たな洞察が得られなくなるまで1つのテーマを掘り下げてから、次へ進む。

## 質問の前に調べること(調べればわかることは聞かない)

各質問の前に、リポジトリの関連箇所を必ず確認する:

- `docs/adr/`(D-xxx)— **既に決定済みの論点を再質問しない**。既存決定と矛盾する提案なら
  「D-xxx を覆すことになるが良いか」という形で質問する
- `docs/ARCHITECTURE.md` — アーキテクチャとの整合
- `docs/GUIDELINE.md` — テスト設計・コーディング規約との整合
- `docs/HANDOVER.md` — プロジェクトの思想・判断基準(原則を崩す提案かどうかの判定に使う)
- 実際のコード(`src/` = フレームワーク、`game/` = サンプルゲーム、`sandbox/` = 疎通確認用)

## このリポジトリで特に突くべき観点

- **層の分離**: そのロジックは純C#層(xUnit でテスト可能)に置けるか。Godot 層に漏れていないか
- **D-013**: フレームワーク(`src/`)の設計にサンプルゲーム(`game/`)の事情を持ち込んでいないか
- **テスタビリティ**: 「失敗するテスト」を先に書ける形か。AI Agent が State 経由で観測・検証できるか
- **待機の規律**: 固定フレーム数・固定秒数の待機を前提にしていないか(State / wait で条件成立を待てるか)
- **決定論**: シード注入などで再現可能に観測できるか
- **後方互換**: 求められていない互換維持でコストを払っていないか

## 質問のルール

- 質問は必ず**一度に一つずつ**行う(AskUserQuestion を使ってよいが、1回1問)
- 各質問には推奨回答を提示する
- 曖昧な回答には具体例・反例で突っ込む

質問フォーマット:

```
### Q[番号]: [質問文]

[なぜこの質問が重要か。関連する D-xxx / コード箇所があれば挙げる]

- **A** — [選択肢]
- **B** — [選択肢]
- ...

**推奨: [A/B/...]** — [理由]
```

## 終了

論点を掘り尽くしたら終了を提案する。続行を求められれば続ける。

終了時のまとめフォーマット:

```
## まとめ

### 決まったこと
- ...

### 記録すべきもの
- 新規/変更の設計判断 → docs/adr/ に D-xxx として記録(決定/背景/トレードオフ/状態)
- アーキテクチャが変わった → docs/ARCHITECTURE.md を更新
- 規約に影響 → docs/GUIDELINE.md を更新
```

まとめで確定した設計判断は、CLAUDE.md の「記録の義務」に従い ADR 化まで提案すること
(ユーザーが同意すれば dig セッションの締めとして D-xxx を起票する)。
